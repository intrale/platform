1. Arquitectura comparada

Modelo A: Intrale

Orquestacion custom: worktrees, 67 hooks Node.js, Telegram bot, dashboard SSE, pipeline multi-skill con 27 skills especializados

Modelo B: Agent Teams

Orquestacion nativa de Claude Code: lider-companeros, lista de tareas compartida, mensajeria inter-agentes, hooks TeammateIdle/TaskCompleted

1.1 Orquestacion de agentes

Intrale: Orquestacion externa

  • Launcher: Start-Agente.ps1 lee sprint-plan.json, crea worktrees, lanza instancias claude CLI
  • Concurrencia: Max 3 agentes. agent-concurrency-check.js promueve de _queue[]
  • Pipeline: Dev→Tester→Builder→Security→Review→QA E2E→Delivery
  • 67 hooks para logging, permisos, monitoreo, notificaciones, gates
  • 27 skills: backend-dev, android-dev, web-dev, qa, tester, review, delivery, planner, scrum, monitor, ops...
  • 3 tiers de modelos: Opus (dev/QA/review), Sonnet (planning/UX), Haiku (ops/delivery)

Agent Teams: Orquestacion nativa

  • Lider: Sesion Claude Code que crea companeros y coordina en lenguaje natural
  • Companeros: Instancias separadas con contexto propio. Recomendado: 3-5
  • Tareas: Lista compartida con file-locking, estados pendiente/progreso/completada
  • Mensajeria: Directa 1:1 y broadcast sin pasar por el lider
  • Hooks: TeammateIdle y TaskCompleted para quality gates
  • Modelos: Configurable por companero (no por fase)

3. Ventajas del modelo actual sobre Agent Teams

Se perderian al migrar:

3.1 Pipeline multi-stage (7 fases)

Dev→Tester→Builder→Security→Review→QA E2E→Delivery, cada uno con modelo optimizado. Agent Teams no tiene concepto de pipeline secuencial por entregable.

3.2 Aislamiento total via worktrees

Elimina conflictos, permite PRs independientes, facilita rollback. Con Agent Teams, 3 companeros en el mismo monorepo = conflictos constantes.

3.3 Monitoreo remoto via Telegram

Lanzar sprint, irse a dormir, monitorear desde celular. Con Agent Teams hay que estar frente a la terminal.

3.4 Dashboard web SSE

6 vistas (overview, flow SVG, activity, roadmap Gantt, CI/CD, logs), refresh cada 5s, screenshots Puppeteer.

3.5 Optimizacion modelos por fase

Opus solo donde hace falta ($15/$75 MTok), Haiku para mecanicos ($0.80/$4 MTok). Ahorro ~35%.

3.6 Auto-recovery

Deteccion zombies/stale, relanzamiento automatico, promocion de cola, cleanup worktrees huerfanos.

3.7 Sprint planning integrado

roadmap.json → /planner → sprint-plan.json → Start-Agente.ps1. Flujo end-to-end automatizado.

3.8 Historial y metricas

agent-metrics.json, sessions-history.jsonl, sprint-audit.jsonl. Alimenta reportes PDF.

4. Ventajas de Agent Teams sobre el modelo actual

Se ganarian al migrar:

4.1 Comunicacion nativa

Mensajeria directa inter-agentes. Valioso para reviews colaborativos, debugging con hipotesis competidoras, "debates" entre agentes.

4.2 Coordinacion dinamica

Auto-claim con dependencias. Redistribucion automatica de trabajo.

4.3 Mantenimiento cero

Feature nativa mantenida por Anthropic. Elimina 67 hooks, scripts, dashboard, bot.

4.4 Hooks evolucionados

4 tipos (command, HTTP, prompt, agent) + eventos TeammateIdle, TaskCompleted, SubagentStart/Stop, WorktreeCreate.

4.5 Aprobacion de plan

Companeros en modo plan (solo lectura) hasta aprobacion del lider. Control de calidad pre-implementacion.

4.6 Interaccion directa

Hablar con cualquier companero individual via Shift+Down o tmux.

4.7 Evolucion futura

Direccion oficial de Anthropic. Features futuras benefician automaticamente.

Costo-efectividad

Intrale es mas caro (~$101) pero justificado en calidad (Opus para razonamiento profundo). Agent Teams con Sonnet es ~9x mas barato pero con calidad inferior. El overhead de hooks Intrale es negligible en tokens (todos son tipo "command", no consumen API).

7. Propuesta de migracion

Estrategia: hibrido incremental

FASE 1 Evaluacion
Simple

Habilitar Agent Teams para investigacion y reviews. 1-2 sprints. Bajo riesgo.

FASE 2 Quality gates nativos
Medio

Hooks TaskCompleted + TeammateIdle. Migrar delivery-gate.js, branch-guard.js. 2-3 sprints. Medio riesgo.

FASE 3 Worktrees para companeros
Grande

CRITICO: WorktreeCreate solo aplica a subagentes, NO a Agent Teams. Requiere soporte de Anthropic. 3-4 sprints. Alto riesgo. BLOQUEANTE.

FASE 4 Monitoreo
Medio

Hooks HTTP + dashboard ligero + Telegram. 2-3 sprints. Medio riesgo.

FASE 5 Sprint planning
Medio

Adaptar /planner para Agent Teams. 2-3 sprints. Medio riesgo.

FASE 6 Corte
Simple

Sprints paralelos comparativos. Deprecar modelo custom si calidad iguala. 2-3 sprints. Bajo riesgo.

8. Recomendacion final

RECOMENDACION: MODELO HIBRIDO

Mantener Intrale para el core del pipeline de desarrollo. Adoptar Agent Teams para tareas complementarias donde la comunicacion inter-agentes agrega valor.

Por que NO migrar completamente

  1. Experimental: Feature flag, sin resume, sin equipos anidados. No es production-ready.
  2. Sin worktrees: Show-stopper para desarrollo paralelo en monorepo.
  3. Sin pipeline: No hay fases. Replicarlo anula la ventaja de simplicidad.
  4. Sin monitoreo remoto: Perder Telegram + dashboard = retroceso operativo.
  5. 12-18 sprints de esfuerzo: Fase 3 bloqueada por Anthropic. ROI no justifica.

Modelo hibrido recomendado

TareaModeloJustificacion
Sprint executionIntralePipeline 7 stages, worktrees, auto-recovery, Telegram
ImplementacionIntraleAislamiento, modelos optimizados, delivery automatizado
QA E2EIntraleDocker, emulador Android, video
Review paraleloAgent Teams3 reviewers con debate
InvestigacionAgent TeamsHipotesis competidoras

Acciones inmediatas (quick wins)

  1. Habilitar CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
  2. Probar con review de PR (3 reviewers paralelos)
  3. Probar con investigacion tecnica multi-perspectiva
  4. Documentar resultados y comparar
  5. Monitorear evolucion: reevaluar cuando Anthropic agregue worktrees + quite flag experimental

Conclusion: El modelo Intrale es un sistema de orquestacion maduro que resuelve problemas reales de produccion. Agent Teams es interesante pero su estado experimental y limitaciones criticas (sin worktrees, sin pipeline, sin monitoreo) lo hacen inadecuado como reemplazo. La estrategia hibrida maximiza el valor de ambos sin riesgos.
