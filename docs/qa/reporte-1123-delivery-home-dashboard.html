1. Resumen Ejecutivo

Las pantallas DeliveryHome y DeliveryDashboard del flavor delivery conforman la operacion diaria del repartidor: ver resumen de pedidos, gestionar pedidos asignados, tomar pedidos disponibles, cambiar estados de entrega y consultar historial. Este reporte documenta 32 casos de prueba (24 funcionales + 8 de seguridad) con:

  • Diagramas de flujo Mermaid por cada flujo y sub-flujo
  • Guion de video paso a paso para cada caso de prueba
  • Criterios de evidencia visual que debe capturar el video
  • Narracion (texto/audio) de cada paso durante la grabacion
  • Documentacion de hallazgos de seguridad pendientes (4 vulnerabilidades)

4. Flujo A — Carga de Home (resumen + activos)

4.1 Descripcion funcional PO

Al abrir la app o volver a Home desde otra pantalla, el ViewModel dispara loadData(). Se cargan en paralelo el resumen de pedidos (contadores) y la lista de pedidos activos asignados al repartidor. La pantalla muestra saludo personalizado, fecha del dia, tres cards de resumen, lista de pedidos activos y un bloque de acciones rapidas con accesos a Dashboard.

Precondiciones: Repartidor logueado con JWT valido | Sesion activa con role=delivery | Conexion a internet | Negocio APPROVED

5. Flujo B — Cambio de Estado de Pedido (inline)

5.1 Descripcion funcional PO

Tanto en Home como en Dashboard, el repartidor puede cambiar el estado de un pedido sin abrir la pantalla de detalle. La accion disponible depende del estado actual: PENDING permite "Iniciar entrega", IN_PROGRESS permite "Marcar entregado" o "Reportar problema". El cambio se hace via PUT al backend, mostrando spinner durante la actualizacion y snackbar al finalizar.

Precondiciones: Sesion activa | Pedido visible en lista | Conexion a internet

5.2 Diagrama de flujo

flowchart TD
  START(("Card visible")) --> CHECK{"order.status"}
  CHECK -->|"PENDING"| BTN_START["Boton 'Iniciar entrega'
(IntraleOutlinedButton)"]
  CHECK -->|"IN_PROGRESS"| BTN_DEL["Boton 'Marcar entregado'
(IntralePrimaryButton)"]
  CHECK -->|"DELIVERED / UNKNOWN"| NO_BTN["Sin boton de accion"]
  BTN_START -->|"Tap"| TAP_S["viewModel.updateStatus(id, IN_PROGRESS)"]
  BTN_DEL -->|"Tap"| TAP_D["viewModel.updateStatus(id, DELIVERED)"]
  TAP_S --> SPIN["state.updatingOrderId == id
CircularProgressIndicator"]
  TAP_D --> SPIN
  SPIN --> HTTP["PUT /{biz}/delivery/orders/{id}/status
{status: 'IN_PROGRESS' o 'DELIVERED'}"]
  HTTP -->|"200 OK"| OK["state.statusUpdateSuccess=true
Snackbar 'Estado actualizado'"]
  HTTP -->|"4xx / 5xx"| ERR["state.statusUpdateError=msg
Snackbar de error"]
  OK --> REFRESH["loadData() -> refresca lista"]
  ERR --> CHECK

  style ERR fill:#ffebee,stroke:#c62828
  style OK fill:#e8f5e9,stroke:#2e7d32
  style SPIN fill:#e3f2fd,stroke:#1565c0

6. Flujo C — Dashboard con Tabs y Filtros

10. Hallazgos de Seguridad (riesgos conocidos)

El Security Agent identifico cuatro vulnerabilidades en el codigo bajo test que NO son nuevas (existen desde antes de este issue de QA compliance) pero deben quedar documentadas como riesgos conocidos. Cada hallazgo requiere un issue de seguridad separado para su remediacion.

SEC-1 (Critica) — Sin validacion de ownership al consultar pedido
Ubicacion: users/src/main/kotlin/ar/com/intrale/DeliveryOrdersFunction.kt:67 + DeliveryOrderRepository:28-31
Descripcion: Cualquier repartidor autenticado puede consultar cualquier pedido por ID dentro del mismo negocio. El metodo repository.getOrder(business, orderId) no compara el email del JWT con el email del repartidor asignado al pedido.
Recomendacion: agregar filtro assignedTo == authenticatedEmail en query DynamoDB, o validar tras el fetch. Si no se cumple, retornar 403/404.
SEC-2 (Critica) — Sin validacion de ownership al cambiar estado/status
Ubicacion: DeliveryOrdersFunction:117 (status), DeliveryOrdersFunction:134 (state)
Descripcion: Repartidor A puede mutar pedidos asignados a Repartidor B. El metodo repository.updateStatus / updateState no valida que el email del JWT coincida con el repartidor asignado al pedido.
Recomendacion: validar ownership antes del update; en caso de mismatch, 403 Forbidden.
SEC-3 (Alta) — DeliveryStateChangeRequest.state es String libre
Ubicacion: DeliveryOrdersFunction:131-134, DeliveryStateDtos.kt
Descripcion: El payload {state: String} acepta cualquier valor sin enum o whitelist. Un repartidor puede enviar "HACKED", "", "DROP TABLE orders".
Recomendacion: usar enum serializable (sealed class o kotlinx.serialization) y validar contra valores permitidos: PENDING, IN_PROGRESS, DELIVERED, NOT_DELIVERED. Rechazar con 400 si no matchea.
SEC-4 (Alta) — Header X-Debug-User como fallback de auth
Ubicacion: DeliveryOrdersFunction:174, DeliveryProfileFunction:115
Descripcion: El metodo resolveEmail(headers) acepta X-Debug-User como fallback cuando no hay JWT valido. En produccion, esto permitiria impersonar a cualquier repartidor solo conociendo su email.
Recomendacion: condicionar el fallback a config.environment == "dev". En PROD/STAGE debe rechazar con 401 si no hay JWT valido. Idealmente eliminar el fallback completo y usar JWTs de testing reales.
