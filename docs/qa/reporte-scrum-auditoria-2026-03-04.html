Problema: Las transiciones Todo→In Progress, In Progress→Ready, Ready→Done no ocurren automáticamente.
Solución: Implementar GitHub Actions que detecten eventos y muevan issues en el board.

Solución: Ejecutar /priorizar sobre los issues huérfanos para clasificar, etiquetar y cerrar los obsoletos.

  • Issues recientes sin board (<15 días): agregar al board en columna apropiada
  • Issue #1205 ya creado para limpiar backlog — ejecutar como próxima prioridad

M3 — ALTO · Resolver PRs antiguos con rama codex/*

Problema: PRs #1031–#1038 usan nomenclatura antigua codex/ (8 PRs con semanas de antigüedad).
Solución: Revisarlos individualmente — merge si están completos, cierre si están obsoletos.

M4 — MEDIO · GitHub Action: detectar issues sin status en board

Problema: Issues se agregan al board sin asignar status (9 encontrados hoy).
Solución: Action semanal que detecte items sin status y los mueva a Backlog Técnico, o notifique via Telegram.

Este issue #1174 ejecutó la corrección manual. La automatización evita que vuelva a ocurrir.

M5 — MEDIO · Usar columna "Refined" para el flujo de refinamiento

La columna "Refined" existe en el board pero nunca se usa. Propuesta: cuando un issue es refinado con /refinar, moverlo de Backlog → Refined. Luego el planner los mueve a Todo al planificar el sprint.

Esto crea un flujo más explícito: Backlog → Refined → Todo → In Progress → Ready → Done

M6 — MEDIO · WIP Limit enforcement

El límite blando de 5 issues en In Progress está definido en methodology.md pero no se aplica. Con la automatización M1, al mover issues a In Progress, el Scrum Master recibirá alerta si se excede el límite.

Propuesta: agregar chequeo de WIP al hook post-git-push.js cuando se crea rama agent/*.

M7 — BAJO · Labels de area para issues UX (#1143–#1148)

Los 6 issues UX movidos hoy a Backlog Técnico solo tienen label ux y enhancement pero les falta area:autenticacion o similar. Sugerencia para /priorizar.
