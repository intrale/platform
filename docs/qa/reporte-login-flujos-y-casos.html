4. Flujo A — Login Basico (email + password)

4.1 Descripcion funcional PO

El usuario registrado y confirmado ingresa su email y contrasena. El sistema valida con Konform en frontend, envia al backend que autentica contra Cognito, verifica que el usuario pertenece al negocio con estado APPROVED, y retorna tokens JWT. Segun el flavor de la app, navega al home correspondiente.

Precondiciones: App instalada (cualquier flavor) | Conexion a internet | Usuario registrado y confirmado en Cognito | Perfil APPROVED en DynamoDB para el negocio

5. Flujo B — Auto-Login (token guardado)

5.1 Descripcion funcional PO

Al abrir la app, el sistema verifica si existe un token JWT guardado en KeyValueStorage. Si existe y es valido (verificado con /validate), el usuario es redirigido automaticamente al Home sin necesidad de ingresar credenciales. Este flujo ejecuta DoCheckPreviousLogin en un LaunchedEffect.

Precondiciones: App instalada | Login previo exitoso con tokens almacenados | Token no expirado en storage

6. Flujo C — Cambio Forzado de Password

6.1 Descripcion funcional PO

Cuando un usuario hace login por primera vez con una contrasena temporal asignada por Cognito, el backend retorna challenge NEW_PASSWORD_REQUIRED. La app muestra campos adicionales (nueva contrasena, nombre y apellido) y reintenta el login con la nueva informacion. El backend ejecuta adminRespondToAuthChallenge y actualiza los atributos del usuario.

Precondiciones: Usuario creado por admin con contrasena temporal | Primer login (Cognito devuelve NEW_PASSWORD_REQUIRED) | App en Login Screen

7. Flujo D — Logout

7.1 Descripcion funcional PO

El usuario autenticado ejecuta logout desde su perfil. El sistema limpia el token JWT, el cache de perfil, el idioma preferido y el SessionStore. Tras el logout, la app navega de vuelta a la pantalla de Login. No hay llamada al backend (es logout local).

Precondiciones: Sesion activa | Usuario en pantalla de Perfil (ClientProfile o DeliveryProfile)
