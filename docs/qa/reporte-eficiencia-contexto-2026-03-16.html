Reporte de Eficiencia: Compactacion de Contexto y Uso de Tokens

Proyecto: Intrale Platform | Fecha: 2026-03-16 | Sprint: SPR-033

Problema: Las sesiones de Claude Code se compactan frecuentemente, perdiendo contexto y requiriendo re-lectura de archivos. La causa raiz es el alto consumo de tokens por skill: 27 skills suman 401 KB de prompts (~100K tokens), y cada agente invoca 6-8 skills por issue, consumiendo ~65% del contexto solo en instrucciones.

Propuesta: Migrar tareas mecanicas (validaciones, checks, builds) de skills-prompt a scripts Node.js/bash, reduciendo el consumo de contexto en ~40% sin perder funcionalidad.

5. Propuesta de optimizacion: Script Pipeline

La propuesta central es crear un pipeline de scripts que ejecute las tareas mecanicas antes y despues de invocar a Claude, reduciendo dramaticamente los tokens consumidos:

Hoy (todo via prompts)

Claude inicia sesion ↓ carga CLAUDE.md (3K tok) ↓ carga Memory (13K tok) ↓ invoca /ops (2.5K tok) ↓ invoca /po (6.5K tok) ↓ invoca /backend-dev (3K tok) ↓ CODEA (variable) ↓ invoca /tester (3K tok) ↓ invoca /security (3.5K tok) ↓ invoca /review (2K tok) ↓ invoca /delivery (5K tok) ↓ invoca /cleanup (7K tok) ═══════════════ OVERHEAD: ~49K tokens COMPACTACIONES: 2-3 por sesion

Propuesto (scripts + prompts)

agent-runner.js (pre-Claude): ✓ ops-check.js (ya existe) ✓ validate-env.js (0 tokens) ✓ git branch setup (0 tokens) Claude inicia sesion ↓ carga CLAUDE.md (3K tok) ↓ carga Memory (13K tok) ↓ invoca /po (6.5K tok) ↓ invoca /dev skill (3K tok) ↓ CODEA (variable) ↓ invoca /review (2K tok) agent-runner.js (post-Claude): ✓ run-tests.js (0 tokens) ✓ security-scan.js (0 tokens) ✓ delivery.js (0 tokens) ✓ cleanup.js (0 tokens) ═══════════════ OVERHEAD: ~28K tokens (-43%) COMPACTACIONES: 0-1 por sesion
Principio: Claude debe hacer lo que solo Claude puede hacer — razonar, analizar, decidir, codear. Todo lo demas (checks, builds, commits, merges, cleanup) debe ser un script que se ejecuta fuera del contexto.

9. Premisa inamovible: sin perdida de funcionalidad

Garantia: Cada script propuesto implementa exactamente la misma logica que el skill que reemplaza. La diferencia es que se ejecuta fuera del contexto de Claude, sin consumir tokens.

Validacion: Cada migracion se verifica con:
  • Test de paridad: skill vs script producen el mismo resultado
  • Dry-run en paralelo: ambos corren, se comparan outputs
  • Rollback inmediato: si el script falla, se invoca el skill original

9.1. Requisito critico: preservar visibilidad de roles en el dashboard

Regla: Cuando un skill se migra a script, el rol del agente debe seguir apareciendo en el dashboard con sus nodos y transiciones, exactamente como si el skill siguiera siendo invocado por Claude.

El dashboard muestra el "Flujo de agentes" como un grafo de nodos (roles) y transiciones (interacciones entre roles). Por ejemplo: BackendDev → Tester → DeliveryManager. Si /tester pasa a ser un script, el nodo "Tester" y sus transiciones no pueden desaparecer del flujo.

Mecanismo: emision de transiciones desde scripts

Hoy, el hook activity-logger.js captura las transiciones automaticamente cuando Claude invoca un skill (detecta el tool call Skill y registra {from, to, ts} en agent_transitions[] de la sesion). Cuando la tarea pasa a ser un script externo, ese hook ya no se dispara.

Solucion: Cada script que reemplaza un skill debe emitir su propia transicion escribiendo directamente en el archivo de sesion activa. Esto se logra con una funcion utilitaria compartida:

emit-transition.js — Modulo utilitario para scripts: // Uso desde cualquier script: // const { emitTransition } = require('./emit-transition'); // emitTransition('BackendDev', 'Tester'); // al iniciar // emitTransition('Tester', 'Security'); // al pasar al siguiente // Internamente: // 1. Lee la sesion activa desde .claude/sessions/ // 2. Agrega {from, to, ts} a agent_transitions[] // 3. Agrega el rol a skills_invoked[] // 4. Escribe de vuelta al archivo de sesion // 5. El dashboard lo lee en el proximo poll → nodo visible

Este modulo se implementa una sola vez y lo usan todos los scripts migrados.

Contratos de cada script migrado

Cada script que reemplaza un skill debe cumplir estos 3 contratos:

#ContratoImplementacion
1 Emitir transicion de entrada emitTransition(prevRole, thisRole) al iniciar
2 Emitir transicion de salida emitTransition(thisRole, nextRole) al terminar
3 Registrar resultado en la sesion Escribir skills_invoked[] + result: pass/fail en el JSON de sesion
Ejemplo concreto: run-tests.js reemplaza a /tester:
1. Emite emitTransition("BackendDev", "Tester")
2. Ejecuta ./gradlew check y parsea resultados
3. Emite emitTransition("Tester", "Security")
4. En el dashboard: nodo "Tester" aparece con icono 🧪, conectado a BackendDev y Security — identico a antes
