Eficiencia de tokens V2 — Determinismo + Splitting

Fecha: 2026-04-22 · Iteración: V2 (profunda) · Proyecto: Intrale Platform · Pipeline V2

Qué cambió respecto del reporte V1 del 2026-04-22 (mañana): El V1 se enfocó en prompt caching + TTS gratis. Este V2 agrega el eje que Leo pidió explícitamente: cuánto de lo que hoy consume tokens puede hacerse SIN modelo (determinismo puro) y cuánto puede bajar de modelo (Opus → Sonnet → Haiku) vía splitting de agentes en fases.

1. Tesis central

El pipeline hoy trata a casi todos los agentes como "unidad monolítica": se spawnea un proceso claude, el rol ya asignado marca un modelo fijo, y el agente hace todo con ese modelo. Pero la mayoría de los skills contienen, en realidad, tres fases muy distintas:

  1. Preparación — recolectar inputs: gh issue view, git diff, lectura de archivos, ejecución de linters, corrida de tests, parseo de reportes XML/JSON. 100% determinístico
  2. Razonamiento — entender el problema, decidir, generar prosa o código nuevo. requiere LLM pesado
  3. Entrega — commit, push, crear PR, subir comentarios, actualizar labels, mover cards. 100% determinístico

Hoy, los tres se pagan al mismo precio (modelo del role). Si partimos el flujo en etapas con el modelo correcto, se evita pagar Opus por hacer gh pr merge.

3. Oportunidades determinísticas fuera de los skills

3.1 Router determinístico en Telegram

Situación actual: el listener (listener-telegram.js) encola todo mensaje al commander, que arranca una sesión claude para resolver incluso comandos triviales. Sin embargo, command-dispatcher.js ya tiene 13+ handlers Node puros (/help, /status, /session, /pendientes, /retry, /limpiar, /detalle, /restart, /reset, /dash-*).

Problema: esos handlers corren dentro de una sesión Claude — el proceso se spawneó aunque el resultado no use el modelo. Eso paga carga del system prompt del Commander (~8K tokens) por nada.

Propuesta: agregar un router determinístico en listener-telegram.js que intercepte comandos reconocidos con regex y los despache a un proceso Node chico (sin spawn de claude). Los mensajes que no hacen match van al commander como hoy.

Ahorro: alto — cada /status hoy gasta al menos 3-5K tokens solo por arrancar la sesión. Leo tipea /status ~8-15 veces/día.

3.2 Compactación del user prompt al spawnear agentes

Hallazgo nuevo crítico. En pulpo.js:3954: 
let userPrompt = `Archivo de trabajo: ...\nContenido:\n${yaml.dump(workData, { lineWidth: -1 })}`;

Se inyecta todo workData sin filtrar. En issues con rebotes, eso incluye histórico completo, logs de builds anteriores, resultados de fases pasadas, comentarios… Fácil 5-20 KB por invocación.

Propuesta: pasar solo paths, no contenidos. El agente lee lo que necesita con Read/cat:

workDataSlim = {
  issue: workData.issue,
  fase: workData.fase,
  pipeline: workData.pipeline,
  rebote: workData.rebote,
  rechazado_en_fase: workData.rechazado_en_fase,
  motivo_rechazo: truncate(workData.motivo_rechazo, 500),
  evidence_paths: {
    build_log: `.pipeline/logs/build-${issue}.log`,
    previous_phase: `.pipeline/procesado/${issue}.*`,
    rebote_history: `.pipeline/rebotes/${issue}.jsonl`
  }
}

Ahorro estimado: ~3-7 KB por spawn × ~200 spawns/semana ≈ 50-100K tokens/semana de input, sin perder información (el agente puede leer bajo demanda).

4. Splitting concreto — 4 casos prioritarios

4.1 /review — el caso más grande

Antes: Opus hace todo: leer PR, diff, evaluar convenciones, análisis semántico, escribir comentarios.

Propuesta (3 fases):

  1. Node review-lint.js: valida convenciones Intrale con regex (logger declarado, statusCode en response, patrón Do, resString, no stringResource fuera de ResStrings). Output: JSON de hallazgos.
  2. Sonnet análisis semántico del diff filtrado (solo archivos con cambios no triviales, excluyendo import reorders y format-only). Lee el JSON de (1) como contexto estructurado.
  3. Opus solo si: diff_loc > 500 OR touches_auth OR touches_payment OR label risk:high.
  4. Node post-proceso: publicar comentarios en GH vía gh api, bajar checks, etiquetar.

Ahorro esperado: ~60% del costo actual de /review.

4.2 /qa — caso multi-etapa

Propuesta:

  1. Node qa-android.sh/qa-api.sh: levantar emulador, instalar APK, correr flow scripted, grabar video, guardar logcat. (Ya existe.)
  2. Node parsear logcat buscando FATAL EXCEPTION, ANR, CRASH. Si no hay → pass automático (no se gasta un solo token).
  3. Sonnet solo si hay fallo: recibir el fragmento relevante del logcat + transcripción simple del video (frames clave con OCR, que es determinístico), decidir causa.
  4. Node edge-tts genera narración del veredicto con template.

Ahorro esperado: en runs sin fallo (la mayoría) → cero LLM. Hoy siempre se ejecuta Opus.

4.3 /tester — chequeos con baseline

Propuesta:

  1. Node correr ./gradlew check, parsear Kover XML, comparar cobertura vs baseline (kover-baseline.json).
  2. Si tests pasan + cobertura ≥ baseline → aprobar sin LLM.
  3. Sonnet solo si hay gap: leer el diff + reportes, proponer 2-5 casos nuevos en template.

Ahorro: ~70% de runs (cuando ya hay tests suficientes) saltan el LLM.

4.4 /security — bajar a Haiku

Propuesta:

  1. Node npm audit, semgrep, detect-secrets, gradle dependencyCheck. Output: JSON estructurado de hallazgos.
  2. Si 0 hallazgos críticos/altos → aprobar sin LLM.
  3. Haiku para triaje de falsos positivos (la info cruda ya está estructurada, no requiere razonamiento complejo).
  4. Node abrir issues de seguridad con template por cada hallazgo confirmado.

Ahorro: pasar de Sonnet a Haiku + salto completo cuando no hay hallazgos → ~80% del costo de /security.

7. Principios de diseño (para no volver a caer en lo mismo)

  1. LLM es para decisiones, no para movimiento de datos. Si una tarea se puede describir como "leer X, aplicar regla Y, escribir Z", no debería tener LLM.
  2. Cuanto más estructurado el input, más barato el modelo que alcanza. Dar a Sonnet un JSON de hallazgos > dar a Opus el log crudo.
  3. El agente no debe recibir lo que puede leer él mismo. Paths > contenidos. Herramientas > dumps.
  4. Pass-through automático cuando no hay señal. Tests verdes + cobertura OK + sin hallazgos = aprobar sin consultar LLM.
  5. Splitting por fase, no por feature. Preparación y entrega son Node; razonamiento es LLM.
