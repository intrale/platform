1. Diagnóstico: por qué fallan las notificaciones

Investigación de Guru reveló 3 causas raíz documentadas en el repo oficial de Claude Code:

1.1 — PermissionRequest se dispara de más (Issue #29212)

El hook PermissionRequest se ejecuta para toda evaluación de permisos, incluyendo las que Claude ya sabe que va a auto-aprobar desde settings. Esto genera 392 invocaciones donde solo 121 requerían acción del usuario.

1.2 — Race condition entre hook y diálogo (Issue #12176)

Si el hook tarda más de ~1.5 segundos en responder, Claude Code muestra el prompt local antes de recibir la respuesta del hook. El usuario ve el prompt en consola aunque el hook ya envió el mensaje a Telegram.

Bug confirmado: Claude Code captura una snapshot de los hooks al iniciar sesión. Si un hook falla repetidamente, puede degradarse internamente. Las modificaciones a hooks mid-session NO se recargan.

1.3 — Notification(permission_prompt) solo se dispara cuando el diálogo SE MUESTRA

Contrario a lo que asumimos, Notification(permission_prompt) se dispara solo cuando el usuario ve el diálogo en consola, NO cuando el hook lo resuelve programáticamente. Esto significa que cuando el approver auto-aprueba rápido (vía hook), la Notification NO se dispara en absoluto.

EventoCuándo se disparaPuede controlar decisión
PermissionRequestEn toda evaluación de permisos (incl. auto-aprobados)Sí: allow/deny
Notification(permission_prompt)Solo cuando el diálogo se muestra al usuario en consolaNo (solo side-effects)
Implicación: Usar Notification como fallback es correcto — solo se dispara cuando el PermissionRequest falló o no corrió, que es exactamente cuando necesitamos alertar.

1.4 — Bug de process.kill(pid, 0) en Windows

La verificación de PID con process.kill(pid, 0) puede dar falsos negativos en Windows. Cuando hay diferencia de permisos entre procesos, lanza EPERM (el proceso EXISTE pero no tenemos permiso para señalizarlo). El código actual trata EPERM como "proceso muerto", lo cual es incorrecto.

// ACTUAL (bug):
try { process.kill(pid, 0); return true; }
catch(e) { return false; }  // EPERM = false ❌

// CORRECTO:
try { process.kill(pid, 0); return true; }
catch(e) { return e.code === "EPERM"; }  // EPERM = true ✅

2. Arquitectura propuesta

2.1 — Principio de diseño: dos canales independientes

En lugar de intentar coordinar hooks frágiles con archivos PID y markers, la arquitectura v2 usa dos canales completamente independientes que convergen en el Commander:

CanalTriggerFunciónFiabilidad
Canal A: Approver PermissionRequest hook Envía mensaje con botones, espera respuesta, controla la decisión ~70% (el hook no siempre se dispara)
Canal B: Watchdog Notification(permission_prompt) + health-check periódico Detecta prompts huérfanos y notifica al usuario ~95% (Notification + health-check como doble red)

2.2 — Flujo completo con fallbacks

Claude Code necesita permiso
    │
    ├── PermissionRequest hook se dispara?
    │   │
    │   ├─ SÍ → permission-approver.js
    │   │       │
    │   │       ├─ Auto-approve? → allow + marker → FIN
    │   │       │
    │   │       ├─ Enviar Telegram OK? → esperar botones → FIN
    │   │       │
    │   │       └─ Telegram falla? → RETRY 1x → si falla → exit(0)
    │   │                                           │
    │   └─ NO ───────────────────────────────┼───┐
    │                                               │   │
    │                                  Claude muestra prompt local
    │                                               │
    │                                  Notification(permission_prompt) se dispara
    │                                               │
    │                                  Canal B: Watchdog
    │                                       │
    │                                       ├─ Approver activo? → ignorar
    │                                       │
    │                                       └─ Sin approver → NOTIFICAR
    │                                           "📵 Permiso pendiente en consola"
    │
    └─ Health-check (cada 10 min): verifica prompts huérfanos
        Si detecta approver muerto con pregunta pending → NOTIFICAR

2.3 — Cambio clave: PID file ANTES del sendMessage

Gap crítico actual: El approver escribe el PID file DESPUÉS de enviar el mensaje (línea 426). Hay una ventana de ~1-3 segundos donde la Notification puede dispararse, no encontrar PID file, y enviar un duplicado.

Fix: Mover la escritura del PID file al inicio del processInput(), antes de cualquier lógica. Si es auto-approve, se borra al salir. Si es Telegram, permanece durante el polling.

2.4 — Endpoint HTTP en el Commander para respuestas instantáneas

Reemplazar el polling de pending-questions.json (150ms) con un endpoint HTTP local en el Commander:

// En telegram-commander.js (ya tiene servidor HTTP en puerto 3100):
// Agregar endpoint:
//   GET /api/decision/:requestId  → long-poll hasta que haya respuesta
//   POST /api/decision/:requestId → registrar decisión (desde callback)

// En permission-approver.js:
// Reemplazar polling de archivo con:
//   fetch("http://localhost:3100/api/decision/" + requestId)
//   → respuesta instantánea cuando el usuario presiona botón

Ventajas: elimina 400 reads/segundo del filesystem, latencia < 5ms, sin race conditions de archivo.

5. Gaps críticos a resolver

🔴 Gap 1: PID file se escribe DESPUÉS del sendMessage

Problema: El approver escribe approver-active.pid en línea 426, DESPUÉS de enviar el mensaje a Telegram. En la ventana de 1-3s entre el envío y la escritura del PID, la Notification puede dispararse, no encontrar PID, y enviar un duplicado.

Fix: Escribir PID file al INICIO de processInput(). Si es auto-approve, borrarlo al salir.

Impacto: Elimina race condition RF-01.4

Esfuerzo: 5 líneas de código

🔴 Gap 2: EPERM tratado como "proceso muerto"

Problema: process.kill(pid, 0) en Windows lanza EPERM cuando el proceso existe pero hay diferencia de permisos. El código actual lo trata como proceso muerto.

Fix: En todos los archivos que usan PID check (notify-telegram.js, commander-launcher.js, health-check.js), cambiar: catch(e) { return e.code === "EPERM"; }

Impacto: Elimina falsos negativos en Windows

Esfuerzo: 3 archivos, 1 línea cada uno

🔴 Gap 3: sendMessage sin retry

Problema: Si el primer sendMessage falla (error transitorio de red, rate limit), el approver hace exit(0) inmediatamente. Un error de 1-2 segundos causa fallback completo.

Fix: Agregar 1 retry con 3s de backoff antes del exit(0).

Impacto: Resuelve FE-02 (Telegram API caída transitoria)

Esfuerzo: 10 líneas

🟡 Gap 4: Botón "Siempre" es texto libre, no botón

Problema: La opción "siempre" solo existe como instrucción de texto. El usuario mobile ve 2 botones y los usa. El concepto "siempre" queda escondido.

Fix: Agregar tercer botón: { text: "✅✅ Siempre", callback_data: "always:" + requestId }

Impacto: UX mejora significativa — menos comandos repetidos

Esfuerzo: 3 líneas (el commander ya soporta always:)

🟡 Gap 5: Callback perdido cuando Commander estaba caído

Problema: Si el Commander cae por >30s (POLL_TIMEOUT_SEC), el callback de Telegram puede descartarse de la cola de getUpdates. El usuario cree haber aprobado pero Claude sigue esperando.

Fix: Al arrancar, el Commander verifica preguntas pending con telegram_message_id y solicita reenvío de respuesta.

Impacto: Resuelve FE-04 (gap crítico del PO)

Esfuerzo: ~30 líneas

🟡 Gap 6: answerCallbackQuery tiene timeout de ~15s

Problema: Si el Commander está ocupado procesando otro comando cuando llega un callback de permiso, puede perder la ventana de 15s para acknowledgearlo.

Fix: Priorizar callbacks de permisos: procesarlos inmediatamente antes de ejecutar comandos largos.

Impacto: Evita "loading..." infinito en Telegram al presionar botón

Esfuerzo: ~20 líneas

Cómo medir

Todas las métricas se calculan desde hook-debug.log. Cada permiso tiene un requestId único con timestamps de: invocación, envío, respuesta, resolución. Un script de auditoría puede calcular las tasas automáticamente.
