4. Flujo 1 — Registro de Negocio (RegisterBusiness)

4.1 Descripcion funcional PO

Un usuario anonimo accede desde la app Business y registra un nuevo negocio completando nombre (minimo 7 caracteres), email del administrador y descripcion. El sistema valida los campos con Konform en frontend y backend, genera un slug unico (normalizacion + sufijo UUID si hay duplicado), persiste el negocio como PENDING en DynamoDB y muestra snackbar de exito.

Precondiciones: App Business instalada | Conexion a internet | Pantalla Login visible

4.3 Campos y validaciones

CampoTipo UIValidacion Frontend (Konform)Validacion Backend (Konform)Mensaje error
nameTextFieldrequired + minLength(1)required + minLength(7)form_error_required / must have at least 7 characters
emailTextFieldrequired + minLength(1) + .+@.+\..+required + .+@.+\..+form_error_invalid_email
descriptionTextFieldrequired + minLength(1)required (not-empty)form_error_required
autoAcceptDeliveriesBoolean (default: false)N/AN/A (optional)N/A
Nota: El frontend valida minLength(1) para name, pero el backend requiere minLength(7). Esto significa que un nombre de 1-6 caracteres pasara validacion frontend pero fallara en backend con error HTTP 400.

4.4 Generacion de slug

Algoritmo slugify():
  1. Convertir a minusculas: value.lowercase()
  2. Normalizar NFD (separar diacriticos): Normalizer.normalize(value, NFD)
  3. Eliminar diacriticos: replace("\\p{InCombiningDiacriticalMarks}+", "")
  4. Reemplazar caracteres no alfanumericos por guion: replace("[^a-z0-9]+", "-")
  5. Eliminar guiones al inicio/final: trim('-')
  6. Si slug ya existe en DB: agregar -{UUID.take(8)}

5. Flujo 2 — Aprobacion / Rechazo (ReviewBusinessRegistration)

5.1 Descripcion funcional PO

Un PlatformAdmin autenticado accede a la pantalla de revision de negocios pendientes. Puede ver la lista de negocios en estado PENDING con sus datos (nombre, descripcion, email admin, autoAcceptDeliveries). Debe ingresar un codigo 2FA valido y luego puede aprobar o rechazar negocios individualmente o en lote. Al aprobar:

  • El estado cambia a APPROVED
  • Si el email admin no existe en Cognito, se crea automaticamente (SignUp)
  • Se asigna el perfil BusinessAdmin al usuario para ese negocio
  • Se calcula el relationState via computeRelationState

Al rechazar, el estado cambia a REJECTED.

Precondiciones: Usuario PlatformAdmin autenticado (JWT) | Al menos un negocio PENDING | Codigo 2FA disponible (TOTP activo)

6. Flujo 3 — Integracion Post-Aprobacion

6.1 Descripcion funcional PO

Tras la aprobacion de un negocio, el sistema ejecuta automaticamente: (1) SignUp del BusinessAdmin en Cognito si no existe, (2) asignacion de perfil BusinessAdmin con computeRelationState, (3) el BusinessAdmin recibe email de Cognito con credenciales temporales y puede hacer login.

Precondiciones: Negocio aprobado via Flujo 2 | Email admin no registrado previamente (para flujo completo)

2 autoAcceptDeliveries no visible en registro: El campo existe en el request pero no hay UI para configurarlo Media Agregar Switch/Checkbox en RegisterNewBusinessScreen o documentar que siempre es false

5 Transaccion no atomica en aprobacion: Si falla el SignUp del BusinessAdmin, el negocio queda en estado inconsistente (no se actualizo aun) Media Evaluar mecanismo de rollback o saga pattern
6 Sin tests Maestro E2E para estos flujos: No existen flujos Maestro automatizados para registro/aprobacion de negocio Media Crear flujos Maestro: register-business.yaml, review-business.yaml
