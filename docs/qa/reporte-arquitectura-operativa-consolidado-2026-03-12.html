1. Visión General: El Ecosistema de Agentes

El sistema operativo de Intrale Platform está diseñado como un ecosistema de agentes orquestados que trabajan juntos de forma proactiva y autónoma:

┌─────────────────────────────────────────────────────────────────┐
│                    ECOSISTEMA DE AGENTES                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Usuario solicita feature/bug fix (CLI o Telegram)             │
│         ↓                                                       │
│  Planner: analiza, prioriza, crea sprint-plan.json             │
│         ↓                                                       │
│  Start-Agente.ps1 lanza 3 agentes máximo en paralelo           │
│         ↓                                                       │
│  Agentes especializados trabajan en worktrees aislados:        │
│    • Agent 1 (issue A) implementa feature                      │
│    • Agent 2 (issue B) implementa bug fix                      │
│    • Agent 3 (issue C) documenta cambios                       │
│         ↓                                                       │
│  Al terminar cada agente:                                       │
│    1. Hook Stop dispara agent-concurrency-check.js             │
│    2. Si hay capacidad, promociona siguiente de _queue[]       │
│    3. Start-Agente.ps1 lanza el nuevo agente en paralelo       │
│         ↓                                                       │
│  Mientras trabajan, sistemas monitorean en background:         │
│    • telegram-commander: espera comandos remotos               │
│    • scrum-monitor-bg: verifica salud del sprint c/30min       │
│    • agent-watcher: detecta agentes terminados                 │
│    • ci-monitor-bg: pollea GitHub Actions tras cada push       │
│         ↓                                                       │
│  Al finalizar todas las historias del sprint:                  │
│    Scrum cierra sprint, genera métricas, notifica velocity     │
└─────────────────────────────────────────────────────────────────┘

2. Ciclo de Vida Completo de un Agente

Fase 1: PLANIFICACIÓN (Planner)

Agente involucrado: /planner

  1. Planner analiza issues abiertos (prioridad, dependencias, tamaño)
  2. Selecciona top N historias accionables (default 7-10)
  3. Valida dependencias con /po (Product Owner)
  4. Crea scripts/sprint-plan.json con estructura:
    • agentes[] — Primeros 3 (lanzados por Start-Agente.ps1)
    • _queue[] — Restantes (4 a N, promovidos automáticamente)
    • _completed[] — Historial de agentes terminados
    • concurrency_limit: 3 — máximo de agentes simultáneos

Fase 2: EJECUCIÓN PARALELA (Agentes especializados)

Agentes involucrados: /backend-dev, /android-dev, /ios-dev, /web-dev, /desktop-dev, /ux, /qa, /security

Cada agente:

  1. Lee su issue con gh issue view <N>
  2. Invoca pipeline de especialistas (/guru → /review → /builder → /tester)
  3. Trabaja en worktree aislado (platform.agent-<issue>-<slug>)
  4. Hace cambios, corre tests, valida build
  5. Invoca /delivery para commit + PR + push
Interacción clave: Los agentes pueden invocar otros agentes en el pipeline:
  • Agent implementa con /backend-dev → invoca /builder para validar → /tester para tests
  • Si falla, agente pide ayuda a /guru para investigar el error
  • Al terminar, /review valida el código antes de /delivery

Fase 3: PROMOCIÓN AUTOMÁTICA

Disparadores:

  • Hook Stop — agent-concurrency-check.js (ejecutado cuando agente termina)
  • Watcher externo — agent-watcher.js (cada 60s, detecta agentes muertos)
Agent 1 termina (git push finalizado)
    ↓
Hook Stop → agent-concurrency-check.js
    ↓
Busca agentes en lista:
  • Cuenta activos (status != "waiting"): 2
  • Límite: 3 → Capacidad disponible: 1 slot
    ↓
¿Hay items en _queue[]?
  ✓ SÍ → Promocionar siguiente de _queue[] a agentes[]
    ↓
Actualizar sprint-plan.json → Lanzar Start-Agente.ps1

     ↓

HORA 10:15 — Agent 1 hace git push
  → post-git-push.js marca "waiting", promociona Agent 4 de _queue
  → ci-monitor-bg inicia polling GitHub Actions
  → AHORA: 4 agentes (1 waiting + 3 activos)

HORA 10:45 — Agent 2 termina
  → Hook Stop: agent-concurrency-check.js
  → Cola vacía → sin promoción

HORA 11:00 — scrum-monitor-bg ejecuta (cada 30min)
  → Verifica salud: Agent 2 terminado, PR sin mergear → Auto-mergea

HORA 11:30 — CI de Agent 1 pasa
  → ci-monitor-bg notifica "CI PASS ✓" → Auto-mergea PR

HORA 12:30 — Último agente termina
  → scrum-monitor-bg cierra sprint → velocity = 100%
  → SPRINT CLOSED → Ready para /planner sprint (SPR-026)

Monitoreo por Subsistema

Agentes (concurrency, promotion, lifecycle)

Qué monitorear: ¿Hay siempre 3 agentes activos? ¿Se promueven de _queue[]? ¿Hay agentes en "waiting"?
Cómo: /monitor en Telegram o cat scripts/sprint-plan.json | jq '.agentes[]'
Auto-reparación: agent-watcher detecta zombies (30+ min) → marca dead → promueve siguiente

CI Monitor (GitHub Actions)

Qué monitorear: ¿Builds se lanzan tras push? ¿Notifica status? ¿Auto-merge funciona?
Cómo: Observar mensajes Telegram: "CI: Testando..." / "CI: PASS ✓" / "CI: FAIL ✗"
Auto-reparación: ci-monitor-bg reintenta polling cada 30s; si falla >5 veces → alerta Telegram

Sprint Health (Project V2)

Qué monitorear: ¿Issues en estado correcto? ¿PRs mergeados = issues Done?
Cómo: /scrum audit en Telegram
Auto-reparación: scrum-monitor-bg detecta inconsistencias cada 30min → auto-repara las menores

Permisos y Seguridad

Qué monitorear: ¿HIGH severity requiere aprobación? ¿Hay intentos bloqueados?
Cómo: tail -50 .claude/hooks/permission-tracker.log
Umbrales: >5 bloqueos en 10min → posible error recurrente

FLUJO A: Agente no se lanza

Pregunta 1: ¿La rama agent/* existe? → NO: /branch <issue> <slug>
Pregunta 2: ¿Hay 3+ agentes activos? → SÍ: Esperar (límite concurrencia)
Pregunta 3: ¿Hay items en _queue[]? → SÍ: Verificar hook Stop → Verificar Start-Agente.ps1 → Lanzar manualmente
Último recurso: /ops --fix + reintentar lanzamiento manual

FLUJO B: Build/Tests fallan

Compilación Kotlin: Leer error → buscar en logs → /guru para investigar → arreglar → /builder
Test assertion: Leer esperado vs actual → /guru → arreglar test O código → /tester
Dependencia Gradle: ./gradlew clean build → verificar red → /guru

FLUJO C: CI/GitHub stuck

Sin PR: Ejecutar /delivery
Workflow no trigger: Verificar .github/workflows/ → re-trigger en web
Workflow stuck >30min: gh run cancel <run-id> + re-run
CI pass pero no merge: gh pr merge <PR> --auto

FLUJO D: Permisos bloqueados

git push main: branch-guard (correcto) → crear rama con /branch
rm -rf: HIGH severity → aprobar en Telegram o usar /cleanup --run
Agente pide comando riesgoso: Denegar → investigar → parar agente

FLUJO E: Telegram no responde

Commander muerto: Matar PID → borrar lock → relanzar commander-launcher.js
Token inválido: Obtener nuevo en @BotFather → actualizar telegram-config.json
Red bloqueada: curl -I https://api.telegram.org/ → verificar VPN/firewall
Chat ID incorrecto: Enviar /start al bot → obtener chat ID correcto

FLUJO F: Sprint desincronizado

PR merged + issue open: node health-check-sprint.js --auto-repair
Issue Done sin PR: Buscar rama → crear PR o revisar historial
Agent done + In Progress: scrum-auto-corrections.js --repair-status
Plan ≠ Project V2: Decidir fuente de verdad → sprint-sync.js --force-sync

FLUJO G-H: Error desconocido / Performance

Error desconocido: /ops → /guru "<error>" → buscar en activity-log → crear issue si persiste
Disco lleno: /cleanup --run --deep
Worktrees antiguos: /cleanup --worktrees --run
Procesos zombie: Identificar con ps aux → matar PIDs viejos
Logs gigantes: /cleanup --logs --run
