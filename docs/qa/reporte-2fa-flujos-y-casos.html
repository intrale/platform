1. Resumen Ejecutivo

La funcionalidad de 2FA (Two-Factor Authentication) comprende 3 flujos principales mas flujos cross-app. El sistema utiliza TOTP (Time-Based One-Time Password) con algoritmo HMAC-SHA1, 6 digitos y ventana de 30 segundos, compatible con apps como Google Authenticator, Authy y similares. Este reporte documenta 37 casos de prueba con:

  • Diagramas de flujo Mermaid por cada flujo y sub-flujo

3.4 Flujo TOTP (Time-Based One-Time Password)

sequenceDiagram
  participant U as Usuario
  participant APP as App Intrale
  participant BE as Backend
  participant COG as Cognito
  participant DB as DynamoDB

  Note over U,DB: Fase 1 - Setup
  U->>APP: Navega a /twoFactorSetup
  APP->>BE: POST /{biz}/2fasetup (JWT)
  BE->>COG: getUser(accessToken)
  COG-->>BE: email, username
  BE->>BE: generateSecret() (SecureRandom + Base32)
  BE->>DB: putItem(User{email, secret})
  BE->>BE: buildOtpAuthUri(secret, email)
  BE-->>APP: TwoFactorSetupResponse{otpAuthUri}
  APP->>U: Deep link o QR/Secreto

  Note over U,DB: Fase 2 - Verify
  U->>APP: Ingresa codigo de 6 digitos
  APP->>BE: POST /{biz}/2faverify (JWT + {code})
  BE->>COG: getUser(accessToken)
  COG-->>BE: email
  BE->>DB: getItem(User{email})
  DB-->>BE: User{secret}
  BE->>BE: TOTP = generate(secret, Instant.now())
  alt code == TOTP
    BE-->>APP: Response(200)
    APP->>U: Snackbar exito
  else code != TOTP
    BE-->>APP: ExceptionResponse("Invalid Two Factor Code")
    APP->>U: Snackbar error
  end

3.5 Formato URI OTP

otpauth://totp/{issuer}:{email}?secret={base32}&issuer={issuer}&algorithm=SHA1&digits=6&period=30
Valores por defecto: issuer="intrale", algorithm=SHA1, digits=6, period=30s
Ejemplo: otpauth://totp/intrale:user@test.com?secret=JBSWY3DPEHPK3PXP&issuer=intrale&algorithm=SHA1&digits=6&period=30

4. Flujo A — Setup 2FA

4.1 Descripcion funcional PO

El usuario autenticado navega a la seccion de seguridad en su perfil y selecciona "Configurar 2FA". El sistema genera un secreto TOTP, lo persiste en DynamoDB vinculado al email, y construye una URI otpauth://. Primero intenta abrir la URI como deep link hacia una app autenticadora. Si el deep link falla, muestra la informacion del secreto con opciones para copiar, compartir y buscar un autenticador.

Precondiciones: Usuario autenticado (JWT valido) | Conexion a internet | Acceso a ClientProfileScreen

5. Flujo B — Verificacion 2FA

5.1 Descripcion funcional PO

El usuario autenticado navega a la seccion de seguridad y selecciona "Verificar 2FA". Ingresa el codigo de 6 digitos generado por su app autenticadora. El frontend valida minLength(6) con Konform, y el backend genera el TOTP esperado con el secreto almacenado y compara. Si coincide, confirma exito; si no, muestra error.

Precondiciones: Usuario autenticado (JWT valido) | 2FA configurado (secret en DynamoDB) | App autenticadora con la cuenta registrada

6. Flujo C — 2FA como Gate en ReviewBusiness

6.1 Descripcion funcional PO

Cuando un PlatformAdmin revisa solicitudes de negocios pendientes, debe ingresar su codigo 2FA junto con la decision (aprobar/rechazar). El backend valida: (1) JWT valido, (2) perfil PlatformAdmin APPROVED, (3) codigo 2FA valido via TwoFactorVerify interno. Solo si las tres validaciones pasan, se ejecuta la operacion. Esto asegura que operaciones sensibles de negocio requieran doble autenticacion.

Precondiciones: PlatformAdmin autenticado (JWT) | Perfil APPROVED | 2FA configurado (secret en DynamoDB) | Al menos un negocio en estado PENDING

6.2 Diagrama de flujo
