3.4 Flujo TOTP (RFC 6238)

sequenceDiagram
  participant U as Usuario
  participant A as App Intrale
  participant B as Backend
  participant C as Cognito
  participant D as DynamoDB
  participant T as TOTP Generator

  Note over U,T: Fase 1 - Setup
  U->>A: Toca "Setup Two Factor"
  A->>B: POST /{biz}/2fasetup (JWT)
  B->>C: getUser(accessToken)
  C-->>B: email
  B->>B: generateSecret() → Base32 (20 bytes)
  B->>D: putItem(email, secret)
  B->>B: buildOtpAuthUri(secret, email, "intrale")
  B-->>A: { otpAuthUri: "otpauth://totp/intrale:email?secret=XXX&..." }
  A->>U: Deep link → App Autenticadora (o UI con QR/secreto)

  Note over U,T: Fase 2 - Verificacion
  U->>A: Ingresa codigo TOTP (6 digitos)
  A->>B: POST /{biz}/2faverify { code: "123456" }
  B->>C: getUser(accessToken)
  C-->>B: email
  B->>D: getItem(email) → secret
  B->>T: generateOTP(secret, now())
  T-->>B: expectedCode
  B->>B: body.code == expectedCode ?
  B-->>A: 200 OK o Error
  A->>U: Snackbar exito o error

4. Flujo A — Setup 2FA (Configuracion inicial)

4.1 Descripcion funcional PO

Usuario autenticado navega a su perfil y activa 2FA. El backend genera un secreto TOTP aleatorio (Base32, 20 bytes), lo persiste en DynamoDB (tabla users), y retorna una URI otpauth://. La app intenta abrir una app autenticadora via deep link. Si falla, muestra UI con secreto enmascarado, opciones de copiar clave/enlace, buscar autenticador y compartir.

Precondiciones: Login exitoso (JWT valido) | App autenticadora instalada (recomendado) | Conexion a internet

5. Flujo B — Verificacion de Codigo TOTP

5.1 Descripcion funcional PO

Usuario con 2FA configurado ingresa el codigo TOTP de 6 digitos generado por su app autenticadora. El backend lee el secreto de DynamoDB, genera el codigo esperado usando TimeBasedOneTimePasswordGenerator (HmacSHA1, 6 digitos, ventana de 30 seg), y compara. Si coincide, retorna 200 OK.

Precondiciones: 2FA configurado (Flujo A completado) | App autenticadora con cuenta intrale | Codigo TOTP vigente (ventana 30 seg) | JWT valido

6. Flujo C — 2FA como Gate en ReviewBusiness

6.1 Descripcion funcional PO

PlatformAdmin revisa solicitudes de registro de negocios. Cada decision (aprobar/rechazar) requiere ingresar un codigo TOTP valido como segundo factor de autenticacion. El backend invoca internamente twoFactorVerify.execute() antes de procesar la decision. Sin 2FA valido, la operacion es rechazada.

Precondiciones: Login como PlatformAdmin | Perfil APPROVED | 2FA configurado | Al menos un negocio PENDING | App autenticadora con codigo vigente
