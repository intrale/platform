1. Resumen Ejecutivo PO

Contexto de negocio

El Dashboard de Negocio es la pantalla principal post-login del BusinessAdmin. Es el punto de entrada al ecosistema operativo del comercio: desde aqui el administrador del negocio accede a sus productos, monitorea pedidos pendientes, gestiona repartidores y configura el negocio. Un administrador puede gestionar multiples negocios — el selector permite cambiar de contexto en tiempo real.

Ola de prioridad 2 — Core de negocio. Esta pantalla es el corazon de la app business. Un fallo aqui bloquea toda la operacion del comercio.

Reglas de negocio aplicables

  • RN-DASH-01: Un BusinessAdmin puede administrar multiples negocios. La combinacion email + business + profile es unica.
  • RN-DASH-02: Si el usuario administra exactamente 1 negocio, se auto-selecciona sin mostrar el selector.
  • RN-DASH-03: Si el usuario administra 2+ negocios, se muestra el dropdown. Si habia una seleccion previa en SessionStore y sigue siendo valida, se restaura.
  • RN-DASH-04: El summary (metricas) solo se carga si hay un negocio seleccionado. Sin negocio → estado MissingBusiness.
  • RN-DASH-05: El endpoint de metricas requiere JWT valido (SecuredFunction). Un token expirado devuelve 401.
  • RN-DASH-06: Los negocios se obtienen via GET /{biz}/searchBusinesses (requiere token).

4. Analisis UX — Evaluacion por Heuristicas

Contexto de uso — BusinessAdmin (H14): Usa la app en el local/cocina, con interrupciones frecuentes. Necesita dashboard rapido, notificaciones de pedidos, gestion batch. Tipicamente en Android, en orientacion portrait, con una mano.

Score UX: 68/100 — Aceptable (necesita mejoras)

#HeuristicaEstadoHallazgos
H1 Visibilidad del estado ⚠️ Parcial Loading usa Text("Cargando negocios...") en vez de skeleton o CircularProgressIndicator. Sin indicador visual de progreso en carga de metricas.
H2 Correspondencia sistema-mundo real ✅ Bien "Panel principal", "Productos", "Pedidos pendientes", "Repartidores activos" — terminologia correcta en espanol argentino.
H3 Control y libertad ⚠️ Parcial Boton "Reintentar" en error de summary: OK. Sin pull-to-refresh para recargar datos manualmente. Sin confirmacion en cambio de negocio (podria generar confusion).
H4 Consistencia y estandares ⚠️ Parcial Cards usan Card(colors=surface) en vez de ElevatedCard para contenido interactivo (segun ux-patterns.md). CTAs usan TextButton en vez de un patron estandar de card action.
H5 Prevencion de errores ✅ Bien Auto-seleccion cuando hay 1 negocio. Validacion de sessionStore antes de usar businessId. No se puede cargar summary sin negocio seleccionado.
H6 Reconocimiento antes que recuerdo ✅ Bien BusinessHeader muestra el nombre del negocio activo. SessionStore restaura seleccion previa. Selector muestra el nombre actual en el campo.
H7 Flexibilidad y eficiencia ⚠️ Parcial Sin pull-to-refresh. Sin atajo para recargar metricas. Sin opcion de ocultar/mostrar tarjetas. Sin acciones batch.
H8 Diseño estetico y minimalista ✅ Bien Layout limpio con cards. Jerarquia tipografica clara (headlineMedium > bodyLarge > titleMedium). Espaciado consistente via MaterialTheme.spacing.
H9 Recuperacion de errores ✅ Bien Error de businesses: mensaje en espanol. Error de summary: mensaje + boton "Reintentar". Estado MissingBusiness con instruccion clara.
H11 Responsividad y adaptabilidad ✅ Bien fillMaxWidth en todas las cards. verticalScroll para pantallas chicas. Espaciado via MaterialTheme.spacing tokens.
H13 Accesibilidad ⚠️ Parcial Sin contentDescription en las DashboardActionCards. TextButton tiene texto — OK. ExposedDropdownMenuBox usa semantica de Material3. Sin semantics {} explícito en las cards.
H14 Contexto de uso por rol ⚠️ Parcial Metricas numericas directas: OK. Sin grafico del dia (ventas). Sin notificacion de pedidos entrantes desde el Dashboard. Para BusinessAdmin en cocina, los botones podrian ser mas grandes.

Friction points detectados

#Pantalla/ComponenteProblemaImpactoEsfuerzo fixPrioridad
FP-01 DashboardScreen — carga inicial Loading mostrado como texto plano en vez de skeleton/spinner. El usuario puede creer que la pantalla se "quedo trabada". MedioSP2
FP-02 DashboardActionCard Usa Card(surface) en vez de ElevatedCard. Segun ux-patterns.md, el contenido interactivo debe usar ElevatedCard para comunicar que es clickable. BajoSP3
FP-03 DashboardScreen — lista de negocios Si el usuario tiene 0 negocios administrados, no hay un empty state dedicado. Cae en summaryState=MissingBusiness con un mensaje poco descriptivo. BajoSP2

FP-05 DashboardActionCard Sin contentDescription en las cards para TalkBack. Usuarios con discapacidad visual no pueden navegar eficientemente. Alto (accesibilidad)SP1

6. Flujo B — Selector de Negocio (multiples negocios)

Escenario: el usuario administra 2+ negocios. Puede cambiar el contexto activo via dropdown.

sequenceDiagram participant U as Usuario participant DS as DashboardScreen participant VM as DashboardViewModel participant SVC as ToGetBusinesses participant SUM as ToGetBusinessDashboardSummary participant SS as SessionStore DS->>VM: loadDashboard() VM->>SVC: execute() SVC-->>VM: Result.success([{id:"biz-001","La Trattoria"}, {id:"biz-002","El Bodegon"}]) VM->>VM: businesses.size > 1, storedBusinessId=null → resolvedBusinessId=null VM->>VM: summaryState = MissingBusiness DS-->>U: Muestra dropdown selector + "Selecciona un negocio para ver las metricas" U->>DS: Abre dropdown, selecciona "El Bodegon" DS->>VM: selectBusiness("biz-002") VM->>SS: updateSelectedBusiness("biz-002") VM->>VM: state.copy(selectedBusinessId="biz-002", summaryState=Loading) DS-->>U: Header: "Negocio: El Bodegon" + "Cargando metricas..." VM->>SUM: execute("biz-002") SUM-->>VM: Result.success({productsCount:8, pendingOrders:5, activeDrivers:1}) DS-->>U: 4 tarjetas actualizadas para "El Bodegon" Note over U,SS: Si el usuario vuelve al Dashboard, SessionStore restaura "biz-002"

Logica de resolucion del negocio seleccionado

Codigo fuente (DashboardViewModel.kt:66-87):
val resolvedBusinessId = when {
  businesses.size == 1 → businesses.first().id
  storedBusinessId != null && businesses.any { it.id == storedBusinessId } → storedBusinessId
  else → null
}

8. Flujo D — Estados de Error y Reintento

stateDiagram-v2 [*] --> BusinessLoading: loadDashboard() BusinessLoading --> BusinessLoaded: API success BusinessLoading --> BusinessError: API failure BusinessLoaded --> SummaryLoading: negocio seleccionado BusinessLoaded --> MissingBusiness: sin negocio seleccionado SummaryLoading --> SummaryLoaded: API success SummaryLoading --> SummaryError: API failure SummaryError --> SummaryLoading: refreshSummary() [reintentar] MissingBusiness --> SummaryLoading: selectBusiness() BusinessError --> BusinessError: sin accion disponible (sin reintentar) SummaryLoaded --> SummaryLoading: selectBusiness() [cambio de negocio]
Gap detectado (PO): El estado BusinessError no tiene boton "Reintentar". Si falla la carga de negocios, el usuario queda bloqueado sin poder refrescar. Se recomienda agregar un CTA de reintento para el error de businesses.
