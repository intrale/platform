3.4 Flujo completo de recuperacion

sequenceDiagram
  participant U as Usuario
  participant APP as App (Compose)
  participant BE as Backend (Ktor)
  participant COG as Amazon Cognito

  U->>APP: Toca "Olvide mi contrasena"
  APP->>APP: Navega a /passwordRecovery
  U->>APP: Ingresa email y toca Recuperar
  APP->>APP: Validacion Konform (email regex)
  APP->>BE: POST /{biz}/recovery {email}
  BE->>BE: Validacion: email required
  BE->>COG: forgotPassword(email)
  COG-->>U: Email con codigo de verificacion
  COG-->>BE: OK
  BE-->>APP: HTTP 200
  APP->>APP: Snackbar + navega a /confirmPasswordRecovery

  U->>APP: Ingresa email + codigo + nueva contrasena
  APP->>APP: Validacion Konform (email, code, password min 8, confirm match)
  APP->>BE: POST /{biz}/confirm {email, code, password}
  BE->>BE: Validacion: email, code, password required
  BE->>COG: confirmForgotPassword(email, code, password)
  COG-->>BE: OK
  BE-->>APP: HTTP 200
  APP->>APP: Snackbar exito + navega a /login (clear backstack)

4. Flujo A — Solicitar Recuperacion de Contrasena

4.1 Descripcion funcional PO

Usuario olvido su contrasena y solicita recuperarla. Ingresa su email en la pantalla PasswordRecovery. El backend llama a cognito.forgotPassword(), que envia un codigo de verificacion al email del usuario. La app muestra un snackbar de exito y navega a ConfirmPasswordRecovery.

Precondiciones: Usuario registrado en Cognito | App en pantalla de Login | Conexion a internet

4.2 Diagrama de flujo

flowchart TD
  START(("Inicio")) --> LOGIN["Pantalla Login"]
  LOGIN -->|"Toca 'Olvide mi contrasena'"| SCREEN["PasswordRecoveryScreen\nCampo: email"]
  SCREEN -->|"Ingresa email"| VAL_FE{"Validacion\nFrontend\n(Konform)"}
  VAL_FE -->|"Email vacio o\nno matchea regex"| ERR_FE["Error inline:\n'Correo invalido'"]
  ERR_FE --> SCREEN
  VAL_FE -->|"Email valido"| LOADING["Boton: spinner +\ndeshabilitado"]
  LOADING --> HTTP["POST /{biz}/recovery\n{email}"]
  HTTP --> VAL_BE{"Validacion\nBackend"}
  VAL_BE -->|"Email vacio/ausente"| E400["HTTP 400\nSnackbar error"]
  VAL_BE -->|"Email valido"| COGNITO["Cognito:\nforgotPassword"]
  COGNITO -->|"NotAuthorizedException"| E401["HTTP 401\nSnackbar error"]
  COGNITO -->|"Otro error"| E500["HTTP 500\nSnackbar error"]
  COGNITO -->|"OK"| OK200["HTTP 200 OK"]
  OK200 --> SNACK["Snackbar:\n'Email de recovery enviado'"]
  SNACK --> CONFIRM["Navega a\n/confirmPasswordRecovery"]

  E401 --> SCREEN

5. Flujo B — Confirmar Recuperacion de Contrasena

5.1 Descripcion funcional PO

Tras solicitar recovery, el usuario recibe un codigo de verificacion por email. En la pantalla ConfirmPasswordRecovery debe ingresar: email, codigo, nueva contrasena y confirmacion de contrasena. El backend llama a cognito.confirmForgotPassword() para establecer la nueva contrasena. Exito navega a Login con backstack limpio.

Precondiciones: Recovery solicitado (Flujo A) | Email recibido con codigo | Codigo no expirado

6. Flujo C — Cambio de Contrasena (Autenticado)

6.1 Descripcion funcional PO

Usuario autenticado desea cambiar su contrasena desde la configuracion de la app. Ingresa su contrasena actual y la nueva. El backend valida el JWT, luego llama a cognito.changePassword() con el token de acceso. Requiere header Authorization con JWT valido.

Precondiciones: Usuario autenticado (sesion activa) | JWT valido en headers | Pantalla accesible desde Settings

7. Flujo D — Cambio Forzado Post-Primer-Login

7.1 Descripcion funcional PO

Cuando un usuario hace login por primera vez (contrasena temporal de Cognito), el backend retorna el desafio NEW_PASSWORD_REQUIRED. La app detecta esto y extiende la pantalla de Login para solicitar: nueva contrasena, nombre y apellido. El backend responde al desafio de Cognito con adminRespondToAuthChallenge y actualiza atributos del usuario.

Precondiciones: Usuario creado administrativamente (contrasena temporal) | Primer login | Desafio NEW_PASSWORD_REQUIRED activo

9.3 Cognito Password Policy

Importante: La password policy de Cognito se configura a nivel del User Pool en AWS y NO esta definida en el codigo del proyecto. El backend NO valida complejidad de contrasena — delega enteramente en Cognito. La configuracion tipica de Cognito incluye:
  • Minimo 8 caracteres
  • Al menos 1 numero
  • Al menos 1 caracter especial
  • Al menos 1 mayuscula
  • Al menos 1 minuscula
Gap identificado: El frontend solo valida minLength(8), por lo que contrasenas que pasan el frontend pueden ser rechazadas por Cognito.
