8. Decisiones arquitectonicas tomadas

8.1 Orden de merge para issues con archivos compartidos

Cuando dos issues tocan el mismo archivo (como worktree-guard.js en #1173 y #1170), el plan define explicitamente el orden de merge. El segundo agente recibe instrucciones de extender en lugar de reemplazar.

8.2 Exclusion de issues con worktree activo

Si un issue ya tiene un worktree creado por un sprint anterior, no se incluye en un nuevo sprint. Esto evita conflictos de rama duplicada en Start-Agente.ps1. Se detecta automaticamente con git worktree list + scan de sibling directories.

8.3 Separacion de responsabilidades: /ops vs /cleanup vs health-check.js

Se identificaron tres niveles de observabilidad del sistema y se mantienen separados por diseno:

  • health-check.js — Hook reactivo (PostToolUse, cooldown 10min). Foco: Telegram commander, approvers, bot. Automatico.
  • /cleanup — Skill de limpieza. Foco: logs, sesiones, worktrees, procesos zombie. Bajo demanda o con --run.
  • /ops — Skill de salud del entorno OS. Foco: JAVA_HOME, PATH, gh CLI, hooks, disco. Preventivo, ideal pre-sprint.
