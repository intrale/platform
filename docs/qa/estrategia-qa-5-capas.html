🛡️ Estrategia de Solución — 5 Capas

Cada capa es independiente y resuelve una causa raíz específica. Se implementan en orden de impacto.

1
Ventanas de QA Exclusivas
Resuelve el 60% de los fallos — gestión inteligente de recursos

Problema

El emulador Android no puede ser persistente 24/7 porque la máquina tiene CPU y RAM limitados. Si el emulador corre en paralelo con agentes de dev o build, el sistema se satura y todo falla. El script qa-android.sh levanta el emulador, pero cuando el agente QA termina o muere, el proceso del emulador muere con él.

Solución — Modo "Ventana de QA"

Cuando se acumulan issues pendientes de QA, el pipeline entra en modo exclusivo de QA:

  • Paso 1: Detectar cola de QA pendiente (threshold configurable, ej: ≥1 issue)
  • Paso 2: Pausar agentes de DEV y BUILD — ambos consumen recursos significativos que colisionan con QA
  • Paso 3: Verificar/generar APK necesario — el build previo debe haber dejado el artefacto en qa/artifacts/
  • Paso 4: Levantar emulador con snapshot qa-ready (~40s boot)
  • Paso 5: Ejecutar todos los QA pendientes en serie (un solo boot del emulador)
  • Paso 6: Al vaciar la cola → apagar emulador → reanudar dev y build

¿Por qué pausar build además de dev?

La fase de build (./gradlew assembleDebug) consume CPU intensivo + hasta 4GB de heap. Si corre en paralelo con el emulador + Maestro + edge-tts, el sistema hace OOM o el emulador se cuelga. Dev y build son las dos fases más costosas y deben pausarse durante la ventana de QA para garantizar estabilidad.

Generación de APK garantizada

La fase de build (previa a QA) debe asegurar que el APK quede disponible como artefacto:

  • Después de un build exitoso → copiar APK a qa/artifacts/composeApp-{flavor}-debug.apk
  • Pre-flight de QA verifica: APK existe, es reciente (<24h), y corresponde al flavor requerido por el issue
  • Si el APK falta o es viejo → rebuild automático antes de abrir la ventana de QA (mientras dev aún puede estar pausado)
  • Esto elimina la re-compilación dentro del agente QA (ahorro de ~5 min y riesgo de OOM)

Implementación técnica

  • Pulpo detecta cola QA y activa ventana en .pipeline/priority-windows.json
  • Flag qa.active = true → pulpo no lanza nuevos agentes dev ni build
  • Flag build.active = true → pulpo pausa también la fase de compilación
  • Script .pipeline/emulator-watchdog.js levanta emulador solo durante la ventana
  • Al vaciar cola → qa.active = false, build.active = false → reanudar pipeline normal
  • Duración dinámica: la ventana se cierra cuando no quedan issues QA pendientes

2
Pre-Flight Checks en Pulpo
Evita desperdiciar tokens en ejecuciones condenadas al fracaso

Problema

El pipeline lanza el agente QA sin verificar que la infraestructura necesaria esté disponible. El agente arranca, descubre que no hay emulador, intenta levantarlo, falla, y consume tokens durante 10-15 minutos antes de reportar qa:failed.

Solución

Agregar 3 verificaciones obligatorias en pulpo.js antes de lanzar el agente QA:

  • ¿Emulador vivo? → adb devices | grep emulator
  • ¿APK disponible? → Verificar existencia en qa/artifacts/
  • ¿Backend responde? → POST a /intrale/signin con body vacío → esperar HTTP 400

Acciones automáticas

  • Si falta algo → reparar automáticamente antes de lanzar, no después
  • Si no se puede reparar → marcar issue como blocked:infra en lugar de qa:failed
  • Evita que el circuit breaker penalice por fallos de infraestructura
3
Separar QA-API de QA-Android
Desbloquea issues de backend que no necesitan emulador

Problema

Issues puramente de backend (API endpoints, funciones Lambda) pasan por el mismo flujo que requiere emulador Android. Quedan bloqueados innecesariamente esperando un emulador que no necesitan.

Solución

Labels del Issue Modo QA Requiere Emulador
area:backend sin app:* QA-API (HTTP requests + validación de respuestas) No
app:client, app:business, app:delivery QA-Android (Maestro flows + video) Sí
area:infra sin app:* qa:skipped con justificación No

Implementación — COMPLETADA

  • Pulpo: preflightQaChecks() clasifica qaMode (android / api / structural) y lo pasa como env var al agente
  • Script: qa/scripts/qa-api.sh — ejecuta test cases del issue como requests HTTP, genera evidencia por criterio de aceptación
  • Generador de test cases: qa/scripts/qa-generate-test-cases.js — genera test cases desde criterios de aceptación del issue como fallback automático
  • Test cases: se generan en etapa de definición (qa/test-cases/{issue}.json). Si faltan, el pre-flight los genera automáticamente. El agente QA solo los consume
  • Rol QA actualizado: .pipeline/roles/qa.md ahora incluye ruteo por QA_MODE con 3 secciones: QA-Android, QA-API, QA Estructural
  • QA-API no pasa por ventana QA: se procesa inmediatamente sin esperar emulador, reduciendo la cola
  • Backend check: QA-API verifica que el backend responda antes de lanzar (reutiliza check 3 de pre-flight)
  • Pre-flight check 5: verifica existencia de test cases; si no existen, ejecuta qa-generate-test-cases.js automáticamente

Spec técnica: docs/qa/qa-api-spec.md

4
Build Produce APK como Artefacto (integrado en Capa 1)
Eliminada como fase independiente — ahora parte del flujo de Ventana QA

Estado: Integrado en Capa 1

Esta capa fue absorbida por la Capa 1 (Ventanas de QA Exclusivas). La generación y verificación del APK ocurre como paso obligatorio antes de abrir la ventana de QA:

  • La fase de build (que corre antes de la ventana QA) debe dejar el APK en qa/artifacts/
  • El pre-flight de Capa 2 verifica que el APK existe, es reciente y del flavor correcto
  • Si falta → rebuild automático mientras dev está pausado pero antes de prender el emulador
  • QA solo instala y testea — nunca compila

Beneficio de la integración

Al estar dentro de la ventana QA (donde build ya está pausado), el rebuild del APK no compite con otros procesos. Se ejecuta con todos los recursos disponibles → más rápido y sin riesgo de OOM.

5
Evidencia con Degradación Gradual
Permite aprobación con evidencia parcial en vez de rechazo total

Problema

El gate de evidencia es all-or-nothing: requiere video >200KB Y audio narrado. Si el emulador falla, no hay video, no hay audio, rechazo completo. No hay fallback.

Solución — Cadena de fallback

Nivel Tipo de Evidencia Resultado
🥇 Ideal Video >200KB + audio narrado (edge-tts) qa:passed
🥈 Aceptable Screenshots secuenciales + logs de API qa:passed (con nota)
🥉 Mínimo Solo logs de API (backend issues) qa:passed (solo para area:backend)
⏸️ Bloqueado Infra no disponible, cambios verificables qa:blocked-env (retry automático)
❌ Rechazo Rama sin cambios de código / criterios inverificables qa:failed

Validación pre-QA de rama

  • Verificar que la rama tiene cambios reales de código Kotlin (no solo config/docs)
  • Si la rama está sincronizada con main sin cambios → rechazar antes de lanzar QA
  • Esto previene el caso del issue #2041 (zero cambios, QA lanzado innecesariamente)

✅ Conclusión

El agente de QA no tiene un problema de lógica ni de código — tiene un problema de infraestructura y resiliencia. La prueba es que cuando la infra está disponible (caso #2019), funciona perfectamente.

La estrategia de 5 capas ataca cada punto de fallo de forma independiente. Las capas 1 y 2 son quick wins que resuelven el 70% del problema con esfuerzo mínimo. Las capas 3-5 consolidan la solución para lograr un pipeline robusto y autónomo.

Con estas 5 capas implementadas, el agente de QA pasa de ser el cuello de botella del pipeline a ser un gate confiable y predecible.
