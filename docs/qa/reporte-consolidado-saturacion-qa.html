2. Causas Raiz de la Saturacion

#CausaSeveridadDetalle
1RAM insuficiente para carga concurrenteCritica16 GB no alcanza para agentes (~5.5 GB) + QA (~7 GB) + sistema (~3 GB) = 15.5 GB sin margen. Un build activo (issue #2017) lo lleva a ~17+ GB.
2Procesos Java residualesAltaGradle daemons y builds previos quedan como huerfanos consumiendo hasta 4 GB. Aunque org.gradle.daemon=false, otros procesos (backend, compilaciones) persisten.
3CPU limitada (4 cores reales)AltaEmulador (2 cores) + Gradle (2-4 cores) + agentes = overcommit severo. Con 3 shards (6 cores) es fisicamente imposible.
4Sin orquestacion de ciclo de vidaMediaNo hay mecanismo para suspender agentes → liberar RAM → ejecutar QA → reanudar. Cada componente se levanta independientemente.
5QA compila redundantementeMediaLa fase QA recompila el APK y/o el backend aunque la fase Build previa ya los genero. Picos de 2.5 GB innecesarios.
6Stack local pesado (Docker + Backend + DB)MediaDynamoDB local + Moto + Backend Ktor consumen ~2.5 GB cuando ya existe infraestructura remota equivalente en AWS.

3.2 Estrategias adoptadas (detalle)

A. Backend Remoto — Emulador pega a Lambda en AWS

Eliminar Docker Desktop, DynamoDB local, Moto/Cognito y el backend Ktor de la maquina. El emulador apunta directamente al API Gateway que ya existe en AWS.

Infraestructura remota existente:

ServicioEndpointEstado
API Gatewayhttps://mgnr0htbvd.execute-api.us-east-2.amazonaws.com/dev/Activo
Lambda (kotlinTest)Detras de API GatewayActivo
DynamoDBus-east-2 (business, users, userbusinessprofile)Activo
CognitoUser Pool en us-east-2Activo

PRO: -2.5 GB RAM; elimina Docker; la app no esta productiva, sin riesgo; usa servicios reales (mas confiable que mocks)

CON: Requiere internet; deploy previo del JAR a Lambda; latencia de red (~100-200ms vs ~5ms, irrelevante para Maestro)

B. Liberacion Pre-QA Orquestada

Antes de iniciar QA: activar QA Priority Window (bloquea nuevos agentes), esperar que builds terminen, matar Java residuales, verificar RAM libre >= 4 GB.

PRO: +3-5 GB RAM; automatizable; sin costo; compatible con cualquier estrategia

CON: Introduce delay (espera de builds); interrumpe trabajo temporalmente

C. APK de la Fase Build (sin recompilar en QA)

La fase Build (agente /builder) ya compila el APK. QA reutiliza ese artefacto directamente en vez de lanzar otro Gradle build de ~2.5 GB.

PRO: Elimina el pico mas grande de consumo; QA arranca inmediatamente sin compilar

CON: Requiere que Build genere los artefactos en ubicacion conocida

3.3 Estrategias descartadas (con justificacion)

D. Reducir footprint del emulador — DESCARTADA

Bajar RAM del emulador de 1536 MB a 1024 MB y cores de 2 a 1.

Razon de descarte: Afecta la fluidez del video (screenrecord) y la estabilidad de los tests Maestro. El ahorro de ~500 MB no justifica la degradacion de calidad de evidencia QA. El emulador se mantiene en 1536 MB / 2 cores.

I. DynamoDB remoto + backend local — DESCARTADA

Mantener backend Ktor local pero apuntando a DynamoDB/Cognito remotos.

Razon de descarte: Si ya vamos a usar AWS remoto, no tiene sentido mantener el backend local (~1.2 GB). Mejor usar Lambda directamente (Estrategia A). Esta opcion solo tendria sentido si el issue modifica codigo de backend y necesitas testearlo antes de deploy — en ese caso el agente de dev ya tiene el backend corriendo en su worktree.

6.2 Opciones de deploy evaluadas

Opcion A — Deploy via GitHub Actions (ELEGIDA)

Concepto: El agente QA dispara el workflow de deploy existente usando gh CLI. GitHub Actions compila el JAR con secrets inyectados y lo deploya a Lambda. Cero consumo de RAM local.

Flujo paso a paso:

# 1. Push de la rama del issue a origin
git push origin agent/2017-fix-saturacion

# 2. Disparar el workflow de deploy apuntando a la rama
gh workflow run main.yml --ref agent/2017-fix-saturacion

# 3. Esperar a que termine (~3-5 min)
while true; do
  STATUS=$(gh run list --workflow=main.yml --limit=1 --json status,conclusion -q '.[0].status')
  if [ "$STATUS" = "completed" ]; then break; fi
  sleep 30
done

# 4. Verificar conclusion
CONCLUSION=$(gh run list --workflow=main.yml --limit=1 --json conclusion -q '.[0].conclusion')
if [ "$CONCLUSION" != "success" ]; then
  echo "ERROR: Deploy fallo"; exit 1
fi

# 5. Health-check: Lambda actualizada y respondiendo
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" \
  -X POST "https://mgnr0htbvd.execute-api.us-east-2.amazonaws.com/dev/intrale/signin" \
  -H "Content-Type: application/json" \
  -d '{"email":"test","password":"test"}')
# Esperamos 400 (Bad Request = Lambda viva, validando input)
if [ "$HTTP_CODE" != "400" ]; then
  echo "WARN: Health-check retorno $HTTP_CODE (esperado 400)"
fi

Que pasa internamente en GitHub Actions:

GitHub Actions Runner (ubuntu-latest, 7 GB RAM): │ ├── 1. Checkout del codigo (rama del issue) ├── 2. Setup JDK 21 (Temurin) + Gradle ├── 3. Inyectar secrets en application.conf: │ sed "s/REGION_VALUE/${REGION}/gi" │ sed "s/ACCESS_KEY_ID/${ACCESS_KEY_ID}/gi" │ sed "s/SECRET_ACCESS_KEY/${ESCAPED_SECRET}/gi" │ sed "s/USER_POOL_ID/${USER_POOL_ID}/gi" │ sed "s/CLIENT_ID/${CLIENT_ID}/gi" ├── 4. ./gradlew build → genera users-all.jar (shadow JAR ~50-80 MB) ├── 5. yvesgurcan/deploy-lambda-function: │ → aws lambda update-function-code │ --function-name kotlinTest │ --zip-file fileb://users/build/libs/users-all.jar │ --region (AWS_DEPLOY_REGION) │ → Credenciales: AWS_DEPLOYER_SECRET_ID / AWS_DEPLOYER_SECRET_KEY └── 6. Lambda kotlinTest actualizada con el codigo de la rama

Cambio requerido en workflow:

# .github/workflows/main.yml — Agregar workflow_dispatch:
on:
  push:
    branches: [ main ]
    paths: [...]
  repository_dispatch:
  workflow_dispatch:          # ← NUEVO: permite disparo desde gh CLI
    inputs:
      deploy_only:            # ← OPCIONAL: para future uso
        type: boolean
        default: false

PRO: Sin AWS CLI local; sin secrets en la maquina; 0 MB de RAM; reutiliza pipeline probado; el JAR se compila en runner con 7 GB RAM

CON: Esperar ~3-5 min al workflow; requiere agregar workflow_dispatch al trigger

Opcion B — Deploy local via AWS CLI — DESCARTADA

Instalar AWS CLI + configurar credenciales + compilar shadowJar localmente + aws lambda update-function-code.

Razon de descarte:

  • Requiere almacenar secrets AWS en disco local (seguridad reducida)
  • El shadowJar consume ~2 GB de RAM para compilar (justamente lo que queremos evitar)
  • Requiere inyectar secrets en application.conf manualmente (sed + archivo de secrets)
  • Setup adicional: instalar AWS CLI, configurar profile intrale-deployer
Opcion C — Hibrido: Build en GitHub + Deploy local — DESCARTADA

GitHub Actions compila el JAR y lo sube como artefacto. QA local lo descarga y lo deploya con AWS CLI.

Razon de descarte: Aun requiere AWS CLI + credenciales locales para el paso de deploy. Si ya se usa GitHub Actions para compilar, mejor que tambien deploya (Opcion A).

7. Restriccion de Paralelismo de QA

Restriccion: Con un solo Lambda remoto compartido (kotlinTest), no se pueden ejecutar dos QA E2E de issues distintos simultaneamente. Los datos de test (usuarios, negocios, estados) se pisarian entre si.

7.1 Mitigacion actual (ya implementada)

  • El pipeline (pulpo.js) ya limita QA a 1 agente secuencial
  • config.yaml: QA usa 1 shard, si 2 agentes llegan juntos uno espera en cola
  • La QA Priority Window garantiza que el agente encolado tiene prioridad sobre nuevos dev
  • En la practica no cambia nada — QA ya era secuencial

7.2 Evolucion futura (si se necesita paralelismo)

OpcionDescripcionComplejidadCuando

Lambda staging fijaCrear kotlinStaging permanente con tablas separadasMedioCuando la app sea productiva
Lambdas efimerasCrear Lambda + tablas temporales por issue, destruir al finalizarGrandeSolo si volumen de QA es alto

Recomendacion: No es necesario ahora. Con el modelo Kanban de 1 QA a la vez y la app no productiva, el backend remoto compartido alcanza perfecto.

12. Resumen Ejecutivo

Problema: 16 GB de RAM es insuficiente para ejecutar QA E2E (~7.4 GB) junto con agentes de desarrollo (~5.5 GB) y el sistema operativo (~3 GB). El sistema se satura y se vuelve impredecible.

Solucion (3 medidas combinadas, costo $0):
  1. Backend remoto: El emulador le pega al Lambda/API Gateway que ya existe en AWS. Elimina Docker + DynamoDB local + Moto + Backend Ktor = -2.5 GB
  2. APK de fase Build: QA no recompila — usa el APK que la fase Build ya genero = -2.5 GB
  3. Liberacion pre-QA: Activar Priority Window + matar residuales antes de arrancar = +3-5 GB libres
Resultado: QA pasa de ~7.4 GB a ~2.4 GB. Reduccion del 68%.

Deploy del backend: Via gh workflow run (GitHub Actions). El workflow existente compila el JAR con secrets inyectados y lo deploya a Lambda. Cero RAM local, cero secrets en la maquina. Solo requiere agregar workflow_dispatch al trigger del workflow.

Emulador: Se mantiene en 1536 MB / 2 cores para garantizar calidad de video y estabilidad de tests.

Paralelismo: QA sigue siendo secuencial (1 issue a la vez). No cambia respecto a hoy. Para el futuro, crear Lambda staging separada cuando la app sea productiva.

Prerequisito critico: Agregar workflow_dispatch a main.yml + crear datos de test en AWS remoto.
