Seccion 1

Arquitectura comparada

Ambos modelos buscan resolver el mismo problema: coordinar multiples agentes IA trabajando en paralelo sobre un codebase. Sin embargo, lo hacen con filosofias radicalmente diferentes.

■ Intrale (modelo actual)

Orquestacion artesanal construida sobre primitivas de SO y git. Cada agente es un proceso claude CLI en un git worktree aislado, gestionado por Start-Agente.ps1 + Run-AgentStream.ps1. La comunicacion ocurre via archivos JSON compartidos, hooks pre/post tool use (~40 hooks), y un dashboard web con SSE.

■ Claude Agent Teams

Orquestacion nativa integrada en Claude Code v2.1.32+. Un lider crea y coordina companeros de equipo con una task list compartida y sistema de mensajeria (buzones). Cada companero tiene su propia ventana de contexto y se comunica directamente con los demas sin pasar por el lider.

Seccion 3

Ventajas del modelo actual sobre Agent Teams

1. Aislamiento total via git worktrees

Cada agente trabaja en una copia completa del repositorio. No existe riesgo de que dos agentes editen el mismo archivo. Agent Teams advierte explicitamente: "Two teammates editing the same file leads to overwrites." En un monorepo como Intrale donde backend, app y tools comparten dependencias, este aislamiento es critico.

2. Pipeline de calidad formalizado y automatizado

El flujo Dev -> Tester -> Review -> QA -> Delivery es ejecutado automaticamente por skills encadenados. Cada gate tiene un skill dedicado con reglas estrictas (SKILL.md), modelo optimizado, y herramientas restringidas. Agent Teams no tiene concepto de pipeline; las task dependencies son secuenciales pero no garantizan gates de calidad.

3. Observabilidad de nivel produccion

El dashboard en localhost:3100 provee: flow graph SVG force-directed, Gantt chart de sprints, metricas KPI en vivo, feed de actividad con SSE cada 5 segundos, tabla de CI/CD, y screenshots automaticos via Puppeteer para Telegram. Agent Teams solo ofrece vista de terminal.

4. Recuperacion autonoma

El agent-watcher.js monitorea agentes muertos, diagnostica la causa, y relanza automaticamente. Incluye retry inteligente cuando un agente pierde su slot de ejecucion. Agent Teams requiere intervencion manual: detectar que un teammate se detuvo, instruir al lider, o generar un reemplazo.

5. Optimizacion de costos por modelo

Cada uno de los 27 skills tiene un modelo asignado en su SKILL.md. Skills simples como /delivery, /review, /tester usan Haiku (barato y rapido). Solo los skills de desarrollo (/backend-dev, /android-dev) usan Opus. Esto es mas granular que la asignacion por teammate de Agent Teams.

6. Integracion con ecosistema externo

Telegram bot para comandos remotos y reportes, GitHub CLI para CI/CD checks, screenshots automaticos para QA evidence, sprint auditing con JSONL persistente. Agent Teams es autocontenido en la terminal sin integraciones externas.

Seccion 4

Ventajas de Agent Teams sobre el modelo actual

1. Comunicacion directa entre agentes

Los teammates se envian mensajes directamente, debaten hipotesis, y se desafian mutuamente. Esto habilita patrones como hipotesis competidoras donde multiples agentes investigan causas raiz y refutan teorias del otro. En Intrale, los agentes son silos: el output de uno se convierte en input del siguiente, pero nunca dialogan.

2. Cero costo de mantenimiento de infraestructura

Agent Teams es una feature nativa de Claude Code. No hay hooks que mantener (~40 archivos JS), no hay dashboard que debuggear (4700 lineas), no hay watcher que supervisar. Las actualizaciones de Claude Code mejoran el sistema automaticamente. En Intrale, cada cambio de API de Claude Code puede romper los hooks.

3. Setup instantaneo con lenguaje natural

Crear un equipo es tan simple como: "Crea un equipo con 3 companeros para refactorizar estos modulos en paralelo." No se necesita conocer PowerShell, git worktrees, ni la estructura de scripts del proyecto. El lider divide el trabajo, asigna tareas, y coordina automaticamente.

4. Task list compartida con coordinacion automatica

La task list usa file locking para prevenir race conditions. Las dependencias entre tareas se resuelven automaticamente: cuando un teammate completa una tarea de la que dependen otras, las tareas bloqueadas se desbloquean. En Intrale, la coordinacion es secuencial y manual (el pipeline define el orden).

5. Aprobacion de plan antes de implementar

Se puede requerir que un teammate planifique en modo read-only antes de implementar. El lider revisa y aprueba el plan. Esto previene re-work costoso. En Intrale, cada agente ejecuta directamente segun su skill sin paso de revision de plan previo.

6. Hooks nativos para quality gates

Los eventos TeammateIdle y TaskCompleted permiten ejecutar scripts de validacion cuando un teammate termina o una tarea se marca como completada. Exit code 2 bloquea la finalizacion y envia feedback. Esto es mas limpio que los 40 hooks custom de Intrale.

Analisis de costo-beneficio

El modelo Intrale es ~50-60% mas barato en tokens API gracias a la optimizacion Haiku/Opus por skill. Sin embargo, el costo oculto de mantenimiento (horas de desarrollo para hooks, dashboard, watcher) no se refleja en los $15-25/sprint. Si se valoriza ese tiempo a $30/hr, el costo real es $135-265/sprint (4-8 horas mantenimiento). Agent Teams elimina ese costo de mantenimiento a cambio de mayor consumo de tokens.

Seccion 7

Propuesta de migracion paso a paso

La recomendacion es una migracion hibrida e incremental. No se trata de reemplazar todo el sistema actual de golpe, sino de incorporar Agent Teams donde aporta valor y mantener las piezas del modelo actual que son superiores.

Fase 1 -- Evaluacion y prueba de concepto
Simple
  • Habilitar CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 en settings.json
  • Ejecutar un sprint de prueba con un issue no critico usando Agent Teams
  • Comparar metricas reales: tokens consumidos, tiempo de ejecucion, calidad del output
  • Documentar limitaciones encontradas en el entorno Windows/MSYS2
  • Validar que CLAUDE.md y skills se cargan correctamente en teammates
Fase 2 -- Agent Teams para tareas de investigacion y review
Simple
  • Usar Agent Teams para code reviews paralelos (seguridad, performance, cobertura)
  • Usar Agent Teams para investigacion de bugs con hipotesis competidoras
  • Mantener el modelo actual para desarrollo e implementacion
  • Crear subagentes reutilizables equivalentes a /review, /security, /perf
Fase 3 -- Hibrido: worktrees + Agent Teams quality gates
Medio
  • Mantener worktrees para aislamiento de desarrollo (ventaja critica)
  • Reemplazar los hooks de quality gate por hooks nativos TaskCompleted y TeammateIdle
  • Migrar la coordinacion de pipeline a task list nativa con dependencias
  • Reducir los ~40 hooks custom a los esenciales (logging, permisos, Telegram)
  • Crear un adapter que convierta el pipeline Dev->Tester->Review->QA->Delivery a task dependencies
Fase 4 -- Migrar observabilidad
Medio
  • Evaluar si el dashboard actual se puede alimentar de la task list nativa de Agent Teams
  • Crear un bridge que lea ~/.claude/teams/ y ~/.claude/tasks/ para el dashboard SSE
  • Mantener integracion Telegram (no existe equivalente en Agent Teams)
  • Agregar metricas de Agent Teams al dashboard existente
Fase 5 -- Modelo nativo con extensiones
Grande
  • Migrar el flujo completo de sprint a Agent Teams cuando la feature salga de experimental
  • Convertir skills criticos (/backend-dev, /android-dev) a subagentes con isolation: worktree
  • Usar isolation: worktree nativo de subagentes para mantener aislamiento git
  • Reemplazar Start-Agente.ps1 por un agente lider que orqueste el sprint completo
  • Mantener: dashboard web (adaptado), Telegram bot, sprint auditing, metricas historicas
  • Eliminar: agent-watcher.js, worktree management manual, hooks de coordinacion

Riesgos de la migracion

  • 1. Agent Teams es experimental: puede cambiar o ser deprecado. No migrar componentes criticos hasta que sea estable.
  • 2. Soporte Windows limitado: split panes no funcionan en Windows Terminal. El modo in-process es la unica opcion viable.
  • 3. Sin resume con teammates: /resume no restaura teammates. Si la sesion se cae, hay que recrear el equipo.
  • 4. Aislamiento de archivos: sin worktrees, teammates pueden sobreescribirse. Mitigacion: usar isolation: worktree en subagentes.
  • 5. Costo de tokens: estimado 2-3x mas caro por sprint. Monitorear con /cost durante la fase de prueba.
Seccion 8

Recomendacion final

Modelo hibrido: mantener las fortalezas, adoptar lo nativo

La recomendacion es un modelo hibrido que preserve las ventajas competitivas del sistema actual (aislamiento via worktrees, pipeline de calidad, observabilidad, recuperacion autonoma) e incorpore selectivamente Agent Teams donde aporta valor genuino (comunicacion inter-agente, coordinacion de tareas, setup simplificado).

MANTENER del modelo actual
  • Git worktrees para aislamiento de codigo (critico para monorepo)
  • Dashboard web con SSE (observabilidad sin equivalente)
  • Integracion Telegram (comandos remotos, reportes)
  • Sprint auditing y metricas historicas
  • Optimizacion Haiku/Opus por skill (ahorro significativo)
  • Pipeline QA con evidencia de video
  • Agent watcher para recovery automatico
ADOPTAR de Agent Teams
  • Comunicacion directa para code reviews en equipo
  • Hipotesis competidoras para debugging complejo
  • Task list nativa con dependencias automaticas
  • Hooks TaskCompleted / TeammateIdle (mas limpios)
  • Aprobacion de plan antes de implementar
  • Subagentes con isolation: worktree cuando sea estable
ELIMINAR gradualmente
  • Hooks de coordinacion que Agent Teams resuelve nativamente
  • Scripts de worktree management cuando isolation: worktree sea estable
  • Hooks de task state que la task list nativa reemplaza
ESPERAR antes de migrar
  • Que Agent Teams salga de estado experimental
  • Soporte adecuado de Windows Terminal
  • Resume/rewind con teammates funcional
  • Evidencia de estabilidad en sprints largos (4-8 horas)

Veredicto ejecutivo

No migrar ahora. El modelo actual de Intrale es mas maduro, mas robusto, y mas barato en tokens que Agent Teams en su estado experimental. Las ventajas de Agent Teams (comunicacion directa, setup simple) no justifican los riesgos (inestabilidad, falta de resume, costo 2-3x mayor, soporte Windows limitado).

Adoptar selectivamente en Fase 1-2 (investigacion y reviews) para ganar experiencia con la herramienta. Preparar la migracion del pipeline para cuando Agent Teams sea GA (Generally Available) y soporte isolation: worktree de forma nativa y estable.

Horizonte estimado para migracion completa: 6-12 meses, condicionado a que Anthropic estabilice la feature y resuelva las limitaciones de Windows y resume/rewind.
