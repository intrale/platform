1. Estado actual del modelo operativo

Como funcionan los agentes hoy

Cada agente es una instancia claude CLI independiente ejecutandose en un git worktree aislado. La orquestacion se realiza via Start-Agente.ps1 que:

  • Lee scripts/sprint-plan.json para obtener las asignaciones
  • Crea worktrees en C:\Workspaces\Intrale\platform.agent-{issue}-{slug}
  • Copia .claude/ del repo principal al worktree
  • Lanza Run-AgentStream.ps1 con pipeline post-Claude automatico
  • Registra PIDs en scripts/sprint-pids.json

Infraestructura existente de estado compartido

sprint-plan.json Plan del sprint: agentes, cola, completados telegram-commander.js Start-Agente.ps1, agent-monitor.js

Diagrama del modelo actual

+-----------------+ +-----------------+ +-----------------+ | Agente 1 | | Agente 2 | | Agente 3 | | (worktree) | | (worktree) | | (worktree) | | /backend-dev | | /android-dev | | /web-dev | +-------+---------+ +-------+---------+ +-------+---------+ | | | | PostToolUse | PostToolUse | PostToolUse v v v +-------+------------------------+------------------------+---------+ | activity-logger.js | | (escribe en REPO_ROOT/.claude/hooks/) | +-------+------------------------+------------------------+---------+ | | | v v v +------------------+ +------------------+ +---------------------+ | agent-registry | | activity-log | | sessions/*.json | | .json | | .jsonl | | | +--------+---------+ +--------+---------+ +----------+----------+ | | | +----------+------------+------------------------+ | v +-----------+-----------+ | dashboard-server.js | | :3100 (SSE + HTML) | +-----------+-----------+ | v +-----------+-----------+ | telegram-commander.js | | (notificaciones) | +-----------------------+ PROBLEMA: No hay flechas horizontales entre Agente 1, 2 y 3. Cada agente opera en aislamiento total.

Limitaciones del modelo actual

  • Sin comunicacion directa: Si Agente 1 (backend-dev) termina una API, Agente 2 (android-dev) no se entera hasta que el PR se mergea.
  • Sin coordinacion de dependencias: Si issue Y depende de issue X, Agente 2 debe esperar pasivamente o fallar al intentar compilar.
  • Handoff manual: El pipeline Dev → Tester → Review → QA se ejecuta secuencialmente, cada etapa relanza un agente nuevo sin contexto previo.
  • Deteccion tardia de conflictos: Dos agentes editando archivos relacionados solo se detecta al momento del merge.

2. Mecanismos nativos de Claude Code reutilizables

Claude Code CLI ofrece varias primitivas que podemos aprovechar para construir comunicacion inter-agente sin reinventar la rueda:

2.1 Agent Teams (Experimental)

Claude Code v2.1.32+ incluye Agent Teams, un sistema experimental de coordinacion multi-agente con:

  • Lista de tareas compartida con estados (pendiente, en progreso, completada) y dependencias
  • Buzon de mensajes nativo para comunicacion directa entre agentes
  • Lider de equipo que coordina, asigna y sintetiza resultados
  • Mensajeria directa: los companieros de equipo pueden enviarse mensajes entre si, no solo al lider

Habilitacion: variable CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 en settings.json o entorno.

Evaluacion para Intrale: Agent Teams asume un lider que coordina y companieros que se generan desde la misma sesion. Nuestro modelo usa worktrees separados lanzados por Start-Agente.ps1, lo cual es incompatible con el modelo de equipos nativos. Sin embargo, los conceptos arquitecturales (buzones, tareas compartidas, mensajeria) son directamente aplicables a nuestra propuesta.

2.2 Hooks reutilizables

Hook Uso para inter-agent Tipo
PostToolUse Ya usado por activity-logger.js. Extender para checkear inbox despues de cada tool call. command
Stop Cuando un agente termina, enviar mensaje de finalizacion al bus. Ya usado por stop-notify.js y agent-concurrency-check.js. command
SessionStart Al iniciar, registrar agente en el bus y leer mensajes pendientes. command
TeammateIdle Disponible en Agent Teams. En nuestro modelo, emulable via Stop + logica custom. command
TaskCompleted Disponible en Agent Teams. En nuestro modelo, emulable cuando un agente marca issue como done. command
SubagentStop Detecta cuando un subagente dentro de una sesion termina. Util para handoffs internos. command

2.3 Subagentes personalizados

Los subagentes (archivos .claude/agents/*.md) permiten crear agentes especializados con:

  • Herramientas restringidas (solo Read/Grep para explorar, o full toolkit para modificar)
  • Modelos especificos (Haiku para tareas rapidas, Sonnet/Opus para complejas)
  • Skills precargados y hooks propios
  • Isolation worktree: el campo isolation: worktree ejecuta el subagente en un worktree temporal
  • Memoria persistente: el campo memory: project permite acumular conocimiento entre sesiones

Aplicacion: Podemos crear un subagente messenger que cada agente invoque para enviar/recibir mensajes, abstrayendo la complejidad del bus.

2.4 Lo que Claude Code NO ofrece (y debemos construir)

  • No hay IPC nativo entre instancias claude CLI separadas. Agent Teams solo funciona dentro del mismo proceso lider.
  • No hay file watcher nativo. Los hooks son event-driven pero ligados al ciclo de tool calls, no a cambios en el filesystem.
  • No hay shared memory. Cada instancia tiene su propio contexto. La unica forma de compartir informacion es via archivos en disco.

Conclusion: Debemos construir un message bus basado en archivos JSON en el directorio compartido .claude/hooks/ del repo principal, con hooks que actuen como event handlers.

3. Arquitectura propuesta: Message Bus Inter-Agente

Diagrama de arquitectura

+-----------------+ +-----------------+ +-----------------+ | Agente 1 | | Agente 2 | | Agente 3 | | /backend-dev | | /android-dev | | /web-dev | | worktree-1 | | worktree-2 | | worktree-3 | +---+----+--------+ +---+----+--------+ +---+----+--------+ | ^ | ^ | ^ | | inbox-check | | inbox-check | | inbox-check | | (PostToolUse) | | (PostToolUse) | | (PostToolUse) v | v | v | +---+----+-------------------+----+-------------------+----+--------+ | | | MESSAGE BUS (archivo JSON) | | REPO_ROOT/.claude/hooks/agent-messages.jsonl | | | | +------------+ +------------+ +------------+ +-------------+ | | | inbox: | | inbox: | | inbox: | | broadcast | | | | agente-1 | | agente-2 | | agente-3 | | (all) | | | +------------+ +------------+ +------------+ +-------------+ | | | +----+----------------------------+----------------------------------+ | | v v +----+--------+ +--------+-----------+ | dashboard | | telegram-commander | | :3100 | | (alertas criticas) | | /messages | +--------------------+ +-------------+

Principios de diseno

  1. Archivos como transporte: Usar JSONL (un mensaje por linea) para evitar parsing de JSON completo en cada lectura.
  2. REPO_ROOT como punto de convergencia: Todos los worktrees resuelven al repo principal via git rev-parse --git-common-dir (patron ya usado en activity-logger.js).
  3. Polling oportunista, no dedicado: Checkear inbox dentro de hooks existentes (PostToolUse) para zero overhead adicional.
  4. Lockfiles para atomicidad: Usar .lock files para escrituras concurrentes, patron ya probado en telegram-commander.js.
  5. Mensajes inmutables: Una vez escrito, un mensaje no se modifica. Los ACKs son mensajes nuevos.
  6. TTL automatico: Mensajes expiran despues de 4 horas (configurable). Cleanup en cada ciclo de polling.

  "id": "msg_1711108200000_backend-dev_001",
  "timestamp": "2026-03-22T14:30:00.000Z",

  "subject": "api-ready",      // tag semantico para filtrar

    "branch": "agent/1780-auth-api",
    "files_changed": ["backend/src/.../SignIn.kt", "backend/src/.../SignUp.kt"],
    "pr_number": 1785
  },

  "ack": false,                // se marca true cuando el receptor lo lee
  "priority": "normal"        // normal | high | critical
}

4.2 Tipos de mensaje

Tipo Descripcion Ejemplo de uso Recipient
request Solicitud que espera respuesta QA pide a Dev que corrija un bug encontrado Agente especifico
response Respuesta a un request previo Dev confirma que corrigio el bug Agente que envio el request
broadcast Mensaje a todos los agentes activos Merge conflict detectado en main *
alert Alerta critica (se reenvia a Telegram) Build roto, tests fallando * + Telegram
signal Senal de coordinacion (Fase 4) Issue X completado, Issue Y puede empezar Agente especifico
handoff Transferencia de contexto (Fase 3) Dev pasa contexto a Tester con resumen del trabajo Agente especifico

4.3 Modulo: agent-messaging.js

// .claude/hooks/agent-messaging.js — Message Bus Inter-Agente
// Modulo compartido: require() desde cualquier hook

const fs = require("fs");
const path = require("path");
const { execSync } = require("child_process");

// Resolver REPO_ROOT (patron de activity-logger.js)
function resolveMainRepoRoot() {
    const candidate = process.env.CLAUDE_PROJECT_DIR || "C:\\Workspaces\\Intrale\\platform";
    try {
        const gitCommon = execSync("git rev-parse --git-common-dir", {
            cwd: candidate, timeout: 3000, windowsHide: true
        }).toString().trim().replace(/\\/g, "/");
        if (gitCommon === ".git") return candidate;
        const gitIdx = gitCommon.indexOf("/.git");
        if (gitIdx !== -1) return gitCommon.substring(0, gitIdx);
        return path.resolve(gitCommon, "..");
    } catch(e) { return candidate; }
}

const REPO_ROOT = resolveMainRepoRoot();
const MESSAGES_FILE = path.join(REPO_ROOT, ".claude", "hooks", "agent-messages.jsonl");
const LOCK_FILE = MESSAGES_FILE + ".lock";
const ACK_FILE = path.join(REPO_ROOT, ".claude", "hooks", "agent-messages-ack.json");
const DEFAULT_TTL_MS = 4 * 60 * 60 * 1000; // 4 horas
const LOCK_TIMEOUT_MS = 3000;
const LOCK_RETRY_MS = 50;

// ─── Lockfile atomico ─────────────────────────────────────────────────────

function acquireLock() {
    const deadline = Date.now() + LOCK_TIMEOUT_MS;
    while (Date.now() < deadline) {
        try {
            fs.writeFileSync(LOCK_FILE, String(process.pid), { flag: "wx" });
            return true;
        } catch (e) {
            // Lock existe: verificar si el proceso dueno sigue vivo
            try {
                const ownerPid = parseInt(fs.readFileSync(LOCK_FILE, "utf8"), 10);
                try { process.kill(ownerPid, 0); } catch {
                    // Proceso muerto: robar lock
                    fs.unlinkSync(LOCK_FILE);
                    continue;
                }
            } catch {}
            // Busy wait
            const start = Date.now();
            while (Date.now() - start < LOCK_RETRY_MS) {}
        }
    }
    return false;
}

function releaseLock() {
    try { fs.unlinkSync(LOCK_FILE); } catch {}
}

// ─── Identificacion del agente actual ─────────────────────────────────────

function getCurrentAgentId() {
    // Determinar agente desde la rama git del worktree
    try {
        const branch = execSync("git rev-parse --abbrev-ref HEAD", {
            cwd: process.env.CLAUDE_PROJECT_DIR || process.cwd(),
            timeout: 3000, windowsHide: true
        }).toString().trim();
        // agent/1780-auth-api → extraer skill del sprint-plan
        const match = branch.match(/^agent\/(\d+)-(.+)$/);
        if (match) {
            return { agent_id: match[2], issue: parseInt(match[1], 10), branch };
        }
    } catch {}
    return { agent_id: "unknown", issue: 0, branch: "main" };
}

// ─── Enviar mensaje ───────────────────────────────────────────────────────

function sendMessage(recipient, type, subject, payload, opts = {}) {
    const sender = getCurrentAgentId();
    const msg = {
        id: `msg_${Date.now()}_${sender.agent_id}_${String(Math.random()).slice(2,6)}`,
        timestamp: new Date().toISOString(),
        type,
        sender: {
            agent_id: sender.agent_id,
            issue: sender.issue,
            skill: opts.skill || "",
            worktree: path.basename(process.env.CLAUDE_PROJECT_DIR || "")
        },
        recipient,
        subject,
        payload: payload || {},
        ttl_ms: opts.ttl_ms || DEFAULT_TTL_MS,
        ack: false,
        priority: opts.priority || "normal"
    };

    if (!acquireLock()) {
        console.error("agent-messaging: no se pudo adquirir lock");
        return null;
    }
    try {
        fs.appendFileSync(MESSAGES_FILE, JSON.stringify(msg) + "\n");
    } finally {
        releaseLock();
    }
    return msg.id;
}

// ─── Leer inbox ──────────────────────────────────────────────────────────

function readInbox(agentId, opts = {}) {
    if (!fs.existsSync(MESSAGES_FILE)) return [];
    const now = Date.now();
    const acks = loadAcks();
    const lines = fs.readFileSync(MESSAGES_FILE, "utf8").split("\n").filter(Boolean);

    return lines
        .map(l => { try { return JSON.parse(l); } catch { return null; } })
        .filter(m => m
            && (m.recipient === agentId || m.recipient === "*")
            && !acks[m.id]
            && (now - new Date(m.timestamp).getTime()) < (m.ttl_ms || DEFAULT_TTL_MS)
        )
        .filter(m => !opts.type || m.type === opts.type)
        .filter(m => !opts.subject || m.subject === opts.subject);
}

// ─── ACK de mensajes ────────────────────────────────────────────────────

function loadAcks() {
    try { return JSON.parse(fs.readFileSync(ACK_FILE, "utf8")); }
    catch { return {}; }
}

function ackMessage(messageId) {
    if (!acquireLock()) return;
    try {
        const acks = loadAcks();
        acks[messageId] = Date.now();
        fs.writeFileSync(ACK_FILE, JSON.stringify(acks));
    } finally {
        releaseLock();
    }
}

// ─── Broadcast ───────────────────────────────────────────────────────────

function broadcast(subject, payload, opts = {}) {
    return sendMessage("*", "broadcast", subject, payload, opts);
}

// ─── Cleanup (llamar periodicamente) ─────────────────────────────────────

function cleanup() {
    if (!fs.existsSync(MESSAGES_FILE)) return;
    if (!acquireLock()) return;
    try {
        const now = Date.now();
        const lines = fs.readFileSync(MESSAGES_FILE, "utf8").split("\n").filter(Boolean);
        const alive = lines.filter(l => {
            try {
                const m = JSON.parse(l);
                return (now - new Date(m.timestamp).getTime()) < (m.ttl_ms || DEFAULT_TTL_MS);
            } catch { return false; }
        });
        fs.writeFileSync(MESSAGES_FILE, alive.join("\n") + (alive.length ? "\n" : ""));

        // Limpiar ACKs de mensajes que ya no existen
        const msgIds = new Set(alive.map(l => {
            try { return JSON.parse(l).id; } catch { return null; }
        }).filter(Boolean));
        const acks = loadAcks();
        for (const k of Object.keys(acks)) {
            if (!msgIds.has(k)) delete acks[k];
        }
        fs.writeFileSync(ACK_FILE, JSON.stringify(acks));
    } finally {
        releaseLock();
    }
}

module.exports = {
    sendMessage, readInbox, ackMessage, broadcast, cleanup,
    getCurrentAgentId, MESSAGES_FILE
};

5.1 Hook inbox-checker.js (PostToolUse)

Se agrega como hook PostToolUse (matcher vacio = todos los tools). Se ejecuta despues de cada tool call y checkea el inbox del agente actual. Si hay mensajes, los inyecta como additionalContext para que Claude los procese naturalmente.

// .claude/hooks/inbox-checker.js — PostToolUse hook
// Checkea inbox del agente y devuelve mensajes como contexto adicional

const messaging = require("./agent-messaging");

const CHECK_INTERVAL_MS = 60000; // No checkear mas seguido que cada 60s
const STATE_FILE = require("path").join(__dirname, "inbox-checker-state.json");

async function main() {
    const input = JSON.parse(require("fs").readFileSync("/dev/stdin", "utf8"));

    // Throttle: no checkear en cada tool call
    let lastCheck = 0;
    try {
        const state = JSON.parse(require("fs").readFileSync(STATE_FILE, "utf8"));
        lastCheck = state.lastCheck || 0;
    } catch {}

    if (Date.now() - lastCheck < CHECK_INTERVAL_MS) {
        process.exit(0); // Demasiado pronto, skip
    }

    // Guardar timestamp
    require("fs").writeFileSync(STATE_FILE, JSON.stringify({ lastCheck: Date.now() }));

    // Determinar identidad
    const agent = messaging.getCurrentAgentId();
    if (agent.agent_id === "unknown") process.exit(0);

    // Leer inbox
    const messages = messaging.readInbox(agent.agent_id);
    if (messages.length === 0) process.exit(0);

    // ACK todos los mensajes leidos
    messages.forEach(m => messaging.ackMessage(m.id));

    // Formatear como contexto para Claude
    const formatted = messages.map(m =>
        `[MENSAJE INTER-AGENTE] De: ${m.sender.agent_id} (issue #${m.sender.issue})\n` +
        `Tipo: ${m.type} | Asunto: ${m.subject} | Prioridad: ${m.priority}\n` +
        `Mensaje: ${m.payload.message || JSON.stringify(m.payload)}`
    ).join("\n---\n");

    // Devolver como additionalContext via JSON stdout
    const output = {
        hookSpecificOutput: {
            hookEventName: "PostToolUse",
            additionalContext: `\n=== MENSAJES RECIBIDOS (${messages.length}) ===\n${formatted}\n` +
                `\nIMPORTANTE: Tenes mensajes de otros agentes. ` +
                `Evalua si alguno requiere accion inmediata (requests, alerts) ` +
                `o si es informativo (broadcasts, signals). ` +
                `Para responder, usa la funcion sendAgentMessage().`
        }
    };

    console.log(JSON.stringify(output));
}

main().catch(e => { console.error(e.message); process.exit(1); });

5.2 Hook stop-messenger.js (Stop)

Se agrega al hook Stop. Cuando un agente termina su turno, envia un broadcast notificando que completó (o fallo).

// .claude/hooks/stop-messenger.js — Stop hook
// Envia broadcast cuando un agente termina

const fs = require("fs");
const messaging = require("./agent-messaging");

async function main() {
    const input = JSON.parse(fs.readFileSync("/dev/stdin", "utf8"));
    const agent = messaging.getCurrentAgentId();
    if (agent.agent_id === "unknown") process.exit(0);

    // Determinar si fue exitoso checkeando si hay PR creado
    let prNumber = null;
    try {
        const { execSync } = require("child_process");
        const prOutput = execSync(
            `gh pr list --head ${agent.branch} --json number --jq ".[0].number"`,
            { timeout: 10000, windowsHide: true, encoding: "utf8" }
        ).trim();
        if (prOutput) prNumber = parseInt(prOutput, 10);
    } catch {}

    // Obtener archivos modificados
    let changedFiles = [];
    try {
        const { execSync } = require("child_process");
        changedFiles = execSync("git diff --name-only origin/main...HEAD", {
            timeout: 5000, windowsHide: true, encoding: "utf8",
            cwd: process.env.CLAUDE_PROJECT_DIR
        }).trim().split("\n").filter(Boolean);
    } catch {}

    messaging.broadcast("agent-completed", {
        message: `Agente ${agent.agent_id} (issue #${agent.issue}) finalizo.`,
        branch: agent.branch,
        pr_number: prNumber,
        files_changed: changedFiles,
        status: prNumber ? "pr-created" : "completed-no-pr"
    });

    // Cleanup periodico (1 de cada 10 veces)
    if (Math.random() < 0.1) messaging.cleanup();
}

main().catch(() => process.exit(0));

6. Fase 3: Handoff de contexto entre agentes

Medio Estimacion: ~400 lineas de codigo

6.1 Concepto de Handoff

Cuando un agente termina su fase del pipeline (Dev → Tester → Review → QA → Delivery), puede "pasar" un resumen estructurado al siguiente agente. Esto elimina la necesidad de que el agente receptor explore todo el PR desde cero.

6.3 Generacion automatica de handoff

El hook stop-messenger.js (Fase 2) se extiende para generar automaticamente el handoff cuando detecta que es un agente Dev que termino exitosamente:

// Extension de stop-messenger.js para handoff automatico

function generateHandoff(agent, prNumber, changedFiles) {
    // Leer el sprint-plan para saber quien es el siguiente en el pipeline
    const planFile = path.join(REPO_ROOT, "scripts", "sprint-plan.json");
    let plan;
    try { plan = JSON.parse(fs.readFileSync(planFile, "utf8")); } catch { return; }

    // Buscar el agente actual en el plan
    const agente = (plan.agentes || []).find(a => a.issue === agent.issue);
    if (!agente) return;

    // Pipeline: dev → tester → review → qa → delivery
    const PIPELINE = ["dev", "tester", "review", "qa", "delivery"];
    const currentStage = agente.skill?.replace("/", "").replace(/-dev$/, "");
    const stageIdx = PIPELINE.indexOf(currentStage);
    if (stageIdx === -1 || stageIdx >= PIPELINE.length - 1) return;

    const nextStage = PIPELINE[stageIdx + 1];

    // Leer ultimo commit message como resumen
    let summary = "";
    try {
        summary = execSync("git log -1 --pretty=%B", {
            cwd: process.env.CLAUDE_PROJECT_DIR,
            timeout: 5000, windowsHide: true, encoding: "utf8"
        }).trim();
    } catch {}

    messaging.sendMessage(nextStage, "handoff", `handoff-${currentStage}-to-${nextStage}`, {
        issue: agent.issue,
        branch: agent.branch,
        pr_number: prNumber,
        summary,
        files_changed: changedFiles.map(f => ({ path: f, action: "modified" })),
        context_for_next: {
            [nextStage]: `Revisar PR #${prNumber} en rama ${agent.branch}. ` +
                         `Archivos modificados: ${changedFiles.length}.`
        }
    });
}

7.3 Hook dependency-checker.js

// .claude/hooks/dependency-checker.js — SessionStart hook
// Verifica si las dependencias del agente estan satisfechas

const fs = require("fs");
const path = require("path");
const messaging = require("./agent-messaging");

const REPO_ROOT = messaging.resolveMainRepoRoot
    ? require("./agent-messaging").REPO_ROOT
    : path.resolve(__dirname, "..", "..");
const PLAN_FILE = path.join(REPO_ROOT, "scripts", "sprint-plan.json");

async function main() {
    const agent = messaging.getCurrentAgentId();
    if (agent.agent_id === "unknown") process.exit(0);

    let plan;
    try { plan = JSON.parse(fs.readFileSync(PLAN_FILE, "utf8")); } catch { process.exit(0); }

    const agente = (plan.agentes || []).find(a => a.issue === agent.issue);
    if (!agente || !agente.depends_on || agente.depends_on.length === 0) process.exit(0);

    // Verificar cada dependencia
    const completados = (plan.completados || []).map(c => c.issue);
    const pending = agente.depends_on.filter(dep => !completados.includes(dep));

    if (pending.length > 0) {
        // Dependencias no satisfechas: informar a Claude
        const output = {
            hookSpecificOutput: {
                hookEventName: "SessionStart",
                additionalContext:
                    `ATENCION: Este issue (#${agent.issue}) tiene dependencias no resueltas:\n` +
                    pending.map(d => `  - Issue #${d} (aun no completado)`).join("\n") +
                    `\n\nOpciones:\n` +
                    `1. Esperar: checkear inbox periodicamente por signal "dependency-resolved"\n` +
                    `2. Trabajar en lo que se pueda sin la dependencia\n` +
                    `3. Enviar request al agente que trabaja en la dependencia pidiendo status`
            }
        };
        console.log(JSON.stringify(output));
    }
}

7.4 Emision de signal cuando una dependencia se resuelve

El agent-concurrency-check.js (ya existente en el hook Stop) se extiende para detectar cuando un agente termina y emitir signals a agentes que dependen de el:

// Extension de agent-concurrency-check.js

function notifyDependents(completedIssue) {
    const plan = JSON.parse(fs.readFileSync(PLAN_FILE, "utf8"));
    const dependents = (plan.agentes || []).filter(a =>
        a.depends_on && a.depends_on.includes(completedIssue)
    );

    for (const dep of dependents) {
        // Verificar si TODAS las dependencias estan satisfechas
        const completados = (plan.completados || []).map(c => c.issue);
        completados.push(completedIssue); // Incluir el que acaba de completar
        const allSatisfied = dep.depends_on.every(d => completados.includes(d));

        messaging.sendMessage(
            dep.slug || String(dep.issue),
            "signal",
            allSatisfied ? "dependency-resolved" : "dependency-progress",
            {
                message: allSatisfied
                    ? `Todas las dependencias de issue #${dep.issue} estan resueltas. Podes avanzar.`
                    : `Issue #${completedIssue} completado (${completados.length}/${dep.depends_on.length} dependencias resueltas).`,
                completed_issue: completedIssue,
                all_satisfied: allSatisfied,
                remaining: dep.depends_on.filter(d => !completados.includes(d))
            },
            { priority: allSatisfied ? "high" : "normal" }
        );
    }
}

9. Riesgos y mitigaciones

Riesgo Severidad Mitigacion
Race conditions en escritura concurrente
Multiples agentes escribiendo al JSONL simultaneamente pueden corromper datos.		 Alta Lockfile con PID + dead-owner detection. El patron ya esta probado en telegram-commander.js (LOCK_FILE con { flag: "wx" }). Ademas, JSONL es append-only, lo que minimiza la ventana de corrupcion. Si el lock falla despues del timeout (3s), el mensaje se descarta silenciosamente.
Overhead de polling en PostToolUse
Cada tool call ejecuta inbox-checker.js, potencialmente desacelerando al agente.		 Media Throttle de 60 segundos. El checker almacena lastCheck en un archivo de estado y hace early return si no pasaron 60s. El costo real es: 1 lectura de estado + 1 comparacion de timestamp = ~1ms. Ademas, el timeout del hook es 5s.
Crecimiento ilimitado del JSONL
Con multiples agentes enviando mensajes, el archivo puede crecer mucho.		 Baja TTL de 4 horas + cleanup periodico. La funcion cleanup() se ejecuta probabilisticamente (10% de las veces) en el hook Stop. Mensajes expirados se eliminan. Estimacion: con 3 agentes activos enviando 1 msg/5min, el JSONL tendra ~144 lineas en 4 horas (~50KB).
Limites de contexto al inyectar mensajes
Muchos mensajes pendientes podrian consumir contexto valioso del agente.		 Media Limitar a 5 mensajes por check. Si hay mas de 5 mensajes pendientes, solo mostrar los 5 mas recientes con prioridad alta primero. El resto se marca como "N mensajes adicionales en inbox".
Mensajes perdidos si un agente muere
Un agente que muere antes de procesar su inbox pierde esos mensajes.		 Baja Los mensajes persisten en el JSONL. Los ACKs son independientes. Si el agente se relanza, leera los mismos mensajes pendientes. El patron es "at-least-once delivery": un mensaje puede leerse mas de una vez, pero nunca se pierde.
Confusion de Claude al recibir mensajes
Claude podria no saber como reaccionar a mensajes inter-agente.		 Media Instrucciones claras en additionalContext. Cada mensaje inyectado incluye instrucciones especificas: "Evalua si requiere accion inmediata". Ademas, extender los skills para incluir instrucciones de como manejar mensajes.
Loops de mensajes
Agente A envia a B, B responde a A, A vuelve a responder, etc.		 Media Rate limit por par sender-recipient. Maximo 3 mensajes del mismo sender al mismo recipient en 10 minutos. Despues, el bus rechaza con warning.

Comparacion: Polling vs File Watcher vs Agent Teams nativo

Enfoque Latencia Complejidad Compatibilidad
Polling en PostToolUse (propuesto) ~60s (throttle) + tempo hasta proximo tool call Baja: reutiliza hooks existentes 100% compatible con worktrees actuales
File watcher (fs.watch) ~1s Media: requiere proceso daemon separado Requiere daemon adicional por worktree
Agent Teams nativo Inmediata (buzon nativo) Alta: requiere redisenar lanzamiento de agentes Incompatible con worktrees separados. Requiere lider+teammates en misma sesion.
HTTP hooks (localhost server) ~100ms Media: requiere server HTTP siempre activo Compatible. Podria usar dashboard-server.js en :3100

Recomendacion

Fase 1-3: Polling en PostToolUse — es la opcion mas simple, no requiere infraestructura adicional, y reutiliza patrones probados. La latencia de ~60s es aceptable para coordinacion de sprint.

Fase 4+: Considerar HTTP hooks via dashboard-server.js. Si la coordinacion de dependencias requiere latencia menor, el dashboard server (:3100) ya corre como daemon. Se podria agregar un endpoint POST /api/messages para push inmediato.
