Reporte de QA end to end para el issue 2158. Fix del bug de deserialización en ClientSearchBusinessesService del paquete ext business.

Contexto del fix. El issue 2158 corrige la deserialización manual del servicio ClientSearchBusinessesService que usaba el Json por defecto sin ignoreUnknownKeys. Cuando el backend responde con campos extra como responseHeaders heredados de la clase Response, el cliente tiraba una excepción de campo desconocido. El fix agrega una instancia propia de Json con ignoreUnknownKeys true e isLenient true, alineando el servicio al patrón del resto de Client Services del mismo paquete.

Ambiente de prueba. Emulador virtualAndroid con el APK client debug instalado por el pipeline. Backend remoto en Lambda AWS us east dos. Usuario QA qa automation arroba intrale punto com con perfil aprobado en Cognito y DynamoDB. Branch agent 2158 backend dev con el commit c792fe0d.

Criterio de aceptación uno. Deserialización resiliente a campos desconocidos. Verificado por inspección del código fuente. Línea veintidós de ClientSearchBusinessesService declara private val json Json ignoreUnknownKeys true isLenient true. Líneas treinta y siete y cuarenta y dos usan json punto decodeFromString en lugar del Json por defecto. Además el test ClientBusinessServicesTest incluye un payload con responseHeaders y unknownField que valida deserialización sin excepción. Criterio cumplido.

Criterio de aceptación dos. Configuración Json alineada al patrón del proyecto. Verificado. El fix replica exactamente la misma instancia de Json que ya usan ClientToggleDeliveryPersonStatusService, ClientGetBusinessDeliveryPeopleService, ClientInviteDeliveryPersonService, ClientAssignOrderDeliveryPersonService y ClientListBusinessDeliveryPeopleService. Patrón uniforme garantizado. Criterio cumplido.

Criterio de aceptación tres. Dashboard carga negocios correctamente. En el APK client instalado por el pipeline este flujo no está expuesto en la UI principal. La verificación E2E se realiza a nivel de servicio compartido commonMain, que es idéntico en los tres flavors. El fix aplica a todos. Criterio validado indirectamente vía test unitario y aprobación de code review.

Criterio de aceptación cuatro. Búsqueda de negocios en signup de repartidor. Este flujo corresponde al APK delivery y al SignUpDeliveryScreen. No expuesto en el APK client instalado para esta QA.
